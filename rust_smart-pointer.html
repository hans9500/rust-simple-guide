01 스택과 힙 메모리 공식 문서 →

Rust의 모든 메모리 관리 개념은 스택과 힙의 차이에서 시작합니다. 어느 메모리 영역에 데이터가 있느냐가 소유권, 복사, 이동의 동작을 결정합니다.

그림 1. 스택과 힙 — 스택 프레임은 LIFO로 쌓이고, 힙은 할당자가 동적으로 관리한다.

스택(Stack)

스택은 컴파일 타임에 크기가 알려진 데이터를 저장합니다. 함수가 호출될 때 프레임(frame)이 쌓이고, 함수가 리턴하면 자동으로 해제됩니다. 포인터를 통한 검색 없이 스택 포인터만 증감하면 되므로 매우 빠릅니다.

  • 정수(i32, u64…), 부동소수(f64…), 불리언(bool), 문자(char)
  • 고정 크기 배열([i32; 5]), 튜플 — 요소 모두 Copy 타입이면 스택에만 존재
  • 참조(&T, &mut T) — 포인터(주소) 자체는 스택, 가리키는 데이터는 힙/스택 모두 가능

힙(Heap)

힙은 런타임에 크기가 결정되거나 크기가 클 때 사용합니다. 할당자(allocator)가 빈 공간을 찾아 포인터를 반환합니다. 스택보다 느리고, 반드시 명시적 해제가 필요합니다. Rust는 소유권 시스템으로 이를 자동화합니다.

  • String — 내부적으로 (ptr, len, capacity) 세 값을 스택에, 실제 문자열 바이트를 힙에 저장
  • Vec<T> — (ptr, len, capacity) 스택 + 실제 요소들 힙
  • Box<T> — 포인터는 스택, 데이터는 힙
fn main() {
    // 스택에만 존재 — Copy 트레이트 구현됨
    let x: i32 = 42;
    let y = x;  // 값 복사 — x도 y도 유효
    println!("x={}, y={}", x, y); // OK

    // 힙을 사용 — Copy 없음, 이동(move) 발생
    let s1 = String::from("hello");
    // s1의 (ptr, len, cap)이 스택에, "hello" 바이트가 힙에 있음
    let s2 = s1;  // s1의 스택 값이 s2로 이동 — 힙 데이터는 그대로
    // println!("{}", s1); // 컴파일 에러! s1은 이동됨
    println!("{}", s2); // OK

    // clone() — 힙 데이터까지 깊은 복사
    let s3 = s2.clone();
    println!("s2={}, s3={}", s2, s3); // 둘 다 유효
}

02 소유권(Ownership) 핵심 규칙 공식 문서 →

Rust의 소유권 시스템은 GC 없이 메모리 안전성을 보장하는 핵심 메커니즘입니다. 세 가지 규칙이 컴파일 타임에 모든 것을 결정합니다.

소유권 세 가지 규칙 (The Rust Book, Ch.4.1)
1. Rust의 각 값은 소유자(owner)가 있다.
2. 한 번에 소유자는 하나만 존재할 수 있다.
3. 소유자가 스코프를 벗어나면 값은 자동으로 해제(drop)된다.

이동(Move) vs 복사(Copy)

기본 타입(i32, bool, f64, char 등)은 Copy 트레이트를 구현하여 대입 시 값이 복사됩니다. Copy가 없는 타입(String, Vec, Box 등)은 대입 시 소유권이 이동(move)되어 원본은 무효화됩니다.

// Copy 타입 — 스택에만 존재, 이동 없이 복사
let a: i32 = 10;
let b = a;  // 비트 단위 복사, a 여전히 유효

// non-Copy 타입 — 이동 발생
let v1 = vec![1, 2, 3];
let v2 = v1;  // v1 → v2 로 소유권 이동
// v1.len(); // error[E0382]: use of moved value: `v1`

// 함수에 전달해도 이동 발생
fn takes_ownership(s: String) {
    println!("{}", s);
} // s가 여기서 drop됨

let my_str = String::from("hello");
takes_ownership(my_str);
// println!("{}", my_str); // 컴파일 에러! 소유권 이전됨

스코프와 Drop

{
    let s = String::from("world"); // s 유효 시작
    println!("{}", s);
} // ← 스코프 끝: drop(s) 자동 호출, 힙 메모리 해제
  // s 여기서 사용 불가

03 Box<T> — 힙에 단독 소유 공식 문서 →

Box<T>는 가장 단순한 스마트 포인터입니다. 스택에는 포인터(8바이트)만, 실제 데이터는 힙에 저장합니다. 소유권 규칙을 그대로 따르며, 스코프를 벗어나면 포인터와 힙 데이터 모두 해제됩니다.

언제 Box<T>를 쓰나?
  • 크기를 컴파일 타임에 알 수 없는 타입 (재귀 타입)
  • 큰 데이터를 힙으로 옮기고 싶을 때
  • 특정 트레이트를 구현하는 값 소유 (Box<dyn Trait>)
  • 단순히 힙에 올리려는 목적만으론 불필요 — Vec/String 써도 됨
메모리 구조
Box::new(5) 생성 시
  • 스택: 포인터 1개 (8 bytes, 64bit)
  • 힙: 정수 5 (4 bytes)
  • Box 드롭 시 힙도 함께 해제
  • Deref로 역참조 → *b == 5

기본 사용

fn main() {
    let b = Box::new(5);
    println!("b = {}", b);   // 자동 역참조 → "b = 5"
    println!("*b = {}", *b); // 명시적 역참조 → "*b = 5"
    // b가 스코프를 벗어나면: 스택의 포인터 + 힙의 5 모두 해제

    // Box는 소유권 규칙 그대로 따름
    let b1 = Box::new(String::from("hello"));
    let b2 = b1; // b1 → b2로 이동 (힙 할당은 그대로, 포인터만 이동)
    // println!("{}", b1); // 에러! b1은 이동됨
    println!("{}", b2); // OK
}

재귀 타입 — Box 없이는 불가능

Rust 컴파일러는 모든 타입의 크기를 컴파일 타임에 알아야 합니다. 재귀 타입은 이론상 무한한 크기가 되므로 컴파일 에러가 납니다. Box를 사용하면 "이 자리는 포인터(고정 8바이트)만 있다"고 알려주어 문제를 해결합니다.

// 에러 예: Box 없이 재귀 타입 정의 시도
// enum List {
//     Cons(i32, List), // error: recursive type has infinite size
//     Nil,
// }

// 해결: Box로 감싸면 Cons 변형 크기 = i32(4) + 포인터(8) = 12 bytes
enum List {
    Cons(i32, Box<List>),
    Nil,
}

use List::{Cons, Nil};

fn main() {
    let list = Cons(1,
        Box::new(Cons(2,
            Box::new(Cons(3,
                Box::new(Nil))))));
}

트레이트 객체 — Box<dyn Trait>

여러 타입을 하나의 컬렉션에 담으려면 컴파일 타임에 크기를 알 수 없습니다. Box<dyn Trait>는 포인터 크기(고정 16바이트 — 데이터 포인터 + vtable 포인터)로 모든 구현체를 균일하게 다룰 수 있게 합니다.

trait Shape {
    fn area(&self) -> f64;
    fn name(&self) -> &str;
}

struct Circle { radius: f64 }
struct Rectangle { w: f64, h: f64 }
struct Triangle { base: f64, height: f64 }

impl Shape for Circle {
    fn area(&self) -> f64 { std::f64::consts::PI * self.radius.powi(2) }
    fn name(&self) -> &str { "원" }
}
impl Shape for Rectangle {
    fn area(&self) -> f64 { self.w * self.h }
    fn name(&self) -> &str { "직사각형" }
}
impl Shape for Triangle {
    fn area(&self) -> f64 { self.base * self.height / 2.0 }
    fn name(&self) -> &str { "삼각형" }
}

fn total_area(shapes: &[Box<dyn Shape>]) -> f64 {
    shapes.iter().map(|s| s.area()).sum()
}

fn main() {
    let shapes: Vec<Box<dyn Shape>> = vec![
        Box::new(Circle { radius: 3.0 }),
        Box::new(Rectangle { w: 4.0, h: 5.0 }),
        Box::new(Triangle { base: 6.0, height: 4.0 }),
    ];

    for s in &shapes {
        println!("{}: {:.2}", s.name(), s.area());
    }
    println!("총 면적: {:.2}", total_area(&shapes));
    // 원: 28.27
    // 직사각형: 20.00
    // 삼각형: 12.00
    // 총 면적: 60.27
}

04 Deref 트레이트와 역참조 강제 공식 문서 →

Deref 트레이트를 구현하면 * 연산자로 역참조할 수 있고, Rust 컴파일러가 자동으로 참조를 변환해주는 역참조 강제(deref coercion)가 작동합니다.

역참조 강제란?

함수/메서드 인자에서 Deref가 구현된 타입을 필요한 참조 타입으로 자동 변환합니다. 이 변환은 컴파일 타임에 일어나므로 런타임 비용이 없습니다.

fn hello(name: &str) {
    println!("Hello, {}!", name);
}

fn main() {
    let m = Box::new(String::from("Rust"));

    // 역참조 강제 체인: Box<String> → &String → &str
    hello(&m);      // &Box<String> → &String → &str 자동 변환
    hello(&*m);     // 같은 의미, 명시적 역참조

    // &str, &String, &Box<String> 모두 hello()에 전달 가능
    let s = String::from("World");
    hello(&s);      // &String → &str 자동 변환
    hello("직접"); // &str 직접 전달

    // Vec<T>도 Deref 구현: &Vec<T> → &[T]
    let v = vec![1, 2, 3];
    let sum: i32 = sum_slice(&v); // &Vec<i32> → &[i32] 자동 변환
    println!("합계: {}", sum);
}

fn sum_slice(s: &[i32]) -> i32 {
    s.iter().sum()
}
역참조 강제 규칙 (Rust Reference 기반)
&T → &U : T가 Deref<Target=U> 구현 시
&mut T → &mut U : T가 DerefMut<Target=U> 구현 시
&mut T → &U : T가 Deref<Target=U> 구현 시 (단방향만 가능)

05 Drop 트레이트 — 자동 메모리 해제 공식 문서 →

Drop 트레이트의 drop 메서드는 값이 스코프를 벗어날 때 컴파일러가 자동으로 호출합니다. C++의 소멸자(destructor)와 동일한 역할이며, 리소스 해제 로직을 여기에 작성합니다.

struct Resource {
    name: String,
}

impl Drop for Resource {
    fn drop(&mut self) {
        println!("리소스 '{}' 해제됨", self.name);
        // 파일 닫기, 락 해제, 네트워크 연결 종료 등
    }
}

fn main() {
    let r1 = Resource { name: String::from("DB 연결") };
    {
        let r2 = Resource { name: String::from("파일 핸들") };
        println!("내부 스코프 종료 직전");
    } // ← r2 drop: "리소스 '파일 핸들' 해제됨" 출력
    println!("main 종료 직전");
} // ← r1 drop: "리소스 'DB 연결' 해제됨" 출력
// 출력 순서: 생성 역순 (LIFO)

// 명시적 조기 해제: drop() 직접 호출은 금지!
// r1.drop(); // error: explicit use of destructor method
// std::mem::drop(r1)은 OK — 소유권을 가져가 즉시 해제
let lock = Resource { name: String::from("Mutex 락") };
std::mem::drop(lock); // 조기 해제 — "리소스 'Mutex 락' 해제됨"
println!("이후에 다른 작업..."); // lock은 이미 해제됨
Drop 호출 순서: 변수는 선언된 역순으로 drop됩니다. 구조체 필드는 선언된 순서대로 drop됩니다. 이는 컴파일러가 보장하며, 이중 해제(double free)는 소유권 시스템이 원천 차단합니다.

06 Rc<T> — 참조 카운팅 공유 소유권 공식 문서 →

Rc<T> (Reference Counted)는 힙에 단일 데이터를 두고 여러 소유자가 공유할 수 있게 합니다. 내부에 강한 참조 카운트(strong_count)를 유지하며, 카운트가 0이 되면 자동으로 데이터를 해제합니다. 단일 스레드에서만 사용 가능합니다 (멀티스레드는 Arc<T> 사용).

왜 단일 스레드만? Rc<T>의 참조 카운트 증감은 원자적(atomic)이지 않습니다. 여러 스레드가 동시에 카운트를 변경하면 데이터 경쟁(data race)이 발생합니다. 멀티스레드 환경에서는 원자적 연산을 사용하는 Arc<T>를 사용하세요.

기본 사용과 참조 카운트

use std::rc::Rc;

fn main() {
    let a = Rc::new(String::from("공유 데이터"));
    println!("a 생성 후 count: {}", Rc::strong_count(&a)); // 1

    let b = Rc::clone(&a); // 힙 데이터 복사 없음! 카운트만 증가
    println!("b 생성 후 count: {}", Rc::strong_count(&a)); // 2

    {
        let c = Rc::clone(&a);
        println!("c 생성 후 count: {}", Rc::strong_count(&a)); // 3
        println!("a={}, b={}, c={}", a, b, c);
    } // c drop → count 2
    println!("c 해제 후 count: {}", Rc::strong_count(&a)); // 2

    // Rc는 불변 참조만 허용 — 여러 곳에서 읽기는 OK
    let len_a = a.len();
    let len_b = b.len();
    println!("a len={}, b len={}", len_a, len_b);
} // a, b drop → count 0 → 힙 데이터 해제

Rc를 이용한 그래프 노드 공유

use std::rc::Rc;

struct Node {
    value: i32,
    children: Vec<Rc<Node>>,
}

fn main() {
    let leaf = Rc::new(Node { value: 3, children: vec![] });
    println!("leaf count: {}", Rc::strong_count(&leaf)); // 1

    let branch = Rc::new(Node {
        value: 2,
        children: vec![Rc::clone(&leaf)], // leaf를 공유 소유
    });
    println!("leaf count: {}", Rc::strong_count(&leaf)); // 2

    let root = Rc::new(Node {
        value: 1,
        children: vec![
            Rc::clone(&branch),
            Rc::clone(&leaf), // leaf를 root도 공유 소유
        ],
    });
    println!("leaf count: {}", Rc::strong_count(&leaf)); // 3
    println!("root value: {}", root.value);
}

07 Weak<T> — 순환 참조 해결 공식 문서 →

Rc<T>만 사용하면 서로를 가리키는 순환 참조(reference cycle)가 발생할 수 있고, 이 경우 카운트가 영원히 0이 되지 않아 메모리 누수가 생깁니다. Weak<T>는 소유권 없이 참조만 보유하여 이 문제를 해결합니다.

순환 참조 예시 — 메모리 누수 발생:
A가 B를 Rc로 가리키고, B가 A를 Rc로 가리키면 → 둘 다 스코프를 벗어나도 카운트가 1씩 남아 절대 해제되지 않음.

순환 참조 발생 예제

use std::rc::Rc;
use std::cell::RefCell;

struct Node {
    value: i32,
    next: Option<Rc<RefCell<Node>>>,
}

fn main() {
    let a = Rc::new(RefCell::new(Node { value: 1, next: None }));
    let b = Rc::new(RefCell::new(Node { value: 2, next: Some(Rc::clone(&a)) }));
    // a.next → b로 만들면 a ↔ b 순환 참조 발생!
    a.borrow_mut().next = Some(Rc::clone(&b));

    // 스코프 종료 시: a count=2, b count=2 → 둘 다 해제 안됨 → 메모리 누수!
    println!("a count: {}", Rc::strong_count(&a)); // 2
    println!("b count: {}", Rc::strong_count(&b)); // 2
}

Weak<T>로 해결 — 트리 부모-자식 관계

use std::rc::{Rc, Weak};
use std::cell::RefCell;

struct TreeNode {
    value: i32,
    // 자식은 Rc — "자식을 소유"
    children: RefCell<Vec<Rc<TreeNode>>>,
    // 부모는 Weak — "부모를 참조만, 소유하지 않음"
    parent: RefCell<Weak<TreeNode>>,
}

fn main() {
    let leaf = Rc::new(TreeNode {
        value: 3,
        children: RefCell::new(vec![]),
        parent: RefCell::new(Weak::new()), // 아직 부모 없음
    });

    let root = Rc::new(TreeNode {
        value: 1,
        children: RefCell::new(vec![Rc::clone(&leaf)]),
        parent: RefCell::new(Weak::new()),
    });

    // leaf의 부모를 root의 Weak 참조로 설정
    *leaf.parent.borrow_mut() = Rc::downgrade(&root);
    // Rc::downgrade → Weak 생성 (strong_count 증가 없음)

    println!("root strong={}, weak={}",
        Rc::strong_count(&root), // 1 — leaf.parent는 weak이라 미포함
        Rc::weak_count(&root),   // 1
    );

    // Weak → Rc 승격: upgrade() → Option<Rc<T>>
    match leaf.parent.borrow().upgrade() {
        Some(p) => println!("leaf의 부모 값: {}", p.value), // 1
        None    => println!("부모 없음"),
    }
} // root drop → strong_count 0 → 해제됨 (순환 참조 없음!)
Weak 사용 원칙: 소유 관계는 Rc, 참조만 필요한 역방향 참조(부모 포인터, 캐시 등)는 Weak로 설계하세요. upgrade()가 None을 반환할 수 있으므로 항상 Option으로 처리해야 합니다.

08 RefCell<T> — 런타임 내부 가변성 공식 문서 →

RefCell<T>는 내부 가변성(Interior Mutability) 패턴을 구현합니다. 컴파일러 대신 런타임에 빌림 규칙을 검사합니다. &self(불변 참조)를 통해서도 내부 데이터를 변경할 수 있습니다.

내부 가변성이란? (std::cell 공식 문서)
"공유 참조(&T)를 통해 내부 상태를 변경할 수 있는 능력." RefCell<T>는 내부적으로 UnsafeCell<T>를 사용하여 컴파일러의 불변성 검사를 우회하고, 대신 런타임 카운터로 안전성을 보장합니다.

RefCell 내부 상태 — borrow_count와 BorrowFlag

RefCell은 내부에 BorrowFlag를 유지합니다. 불변 빌림 시 양수로 증가, 가변 빌림 시 음수 플래그로 전환됩니다. 규칙 위반 시 즉시 패닉합니다.

use std::cell::RefCell;

fn main() {
    let data = RefCell::new(vec![1, 2, 3]);

    // ── 케이스 1: 불변 빌림 여러 개 동시 → OK ──
    {
        let r1 = data.borrow(); // Ref<Vec<i32>>, flag = +1
        let r2 = data.borrow(); // flag = +2
        println!("r1={:?}, r2={:?}", *r1, *r2);
    } // r1, r2 drop → flag = 0

    // ── 케이스 2: 가변 빌림 → OK ──
    {
        let mut w = data.borrow_mut(); // RefMut, flag = -1
        w.push(4);
        w.push(5);
    } // w drop → flag = 0
    println!("{:?}", data.borrow()); // [1, 2, 3, 4, 5]

    // ── 케이스 3: 패닉 방지 — try_borrow / try_borrow_mut ──
    let _guard = data.borrow(); // 불변 빌림 활성 중
    match data.try_borrow_mut() {
        Ok(mut w) => w.push(99),
        Err(e) => println!("빌림 실패 (패닉 없음): {}", e),
        // "already borrowed: BorrowMutError"
    }
}

내부 가변성 패턴 — &self 메서드 안에서 변경

use std::cell::RefCell;

// 피보나치 수열 메모이제이션 캐시
struct FibCache {
    cache: RefCell<Vec<u64>>,
}

impl FibCache {
    fn new() -> Self {
        FibCache { cache: RefCell::new(vec![0, 1]) }
    }

    // &self — 외부에서 보면 불변이지만 내부적으로 캐시를 채움
    fn get(&self, n: usize) -> u64 {
        if n < self.cache.borrow().len() {
            return self.cache.borrow()[n];
        }
        let mut c = self.cache.borrow_mut();
        while c.len() <= n {
            let len = c.len();
            let next = c[len - 1] + c[len - 2];
            c.push(next);
        }
        c[n]
    }
}

fn main() {
    let fib = FibCache::new(); // mut 없이 선언!
    println!("fib(10) = {}", fib.get(10)); // 55
    println!("fib(20) = {}", fib.get(20)); // 6765
    println!("fib(10) = {}", fib.get(10)); // 55 (캐시 히트)

    // &self 이므로 동시에 여러 참조자도 가능
    let r1 = &fib;
    let r2 = &fib;
    println!("r1.get(5)={}, r2.get(6)={}", r1.get(5), r2.get(6));
}

09 Rc<RefCell<T>> — 공유 + 내부 가변 공식 문서 →

Rc<T>는 공유 소유권을, RefCell<T>는 내부 가변성을 제공합니다. 둘을 조합하면 여러 소유자가 같은 데이터를 공유하면서 각자 변경할 수 있습니다. 그래프, GUI 위젯 트리, 옵저버 패턴 등에서 필수 패턴입니다.

옵저버 패턴 — 실전 예제

use std::rc::Rc;
use std::cell::RefCell;

// 공유 상태
struct EventBus {
    listeners: Vec<Box<dyn Fn(&str)>>,
}

impl EventBus {
    fn new() -> Self { EventBus { listeners: vec![] } }

    fn subscribe(&mut self, f: impl Fn(&str) + 'static) {
        self.listeners.push(Box::new(f));
    }

    fn publish(&self, event: &str) {
        for listener in &self.listeners {
            listener(event);
        }
    }
}

struct Widget {
    name: String,
    bus: Rc<RefCell<EventBus>>,
}

impl Widget {
    fn new(name: &str, bus: Rc<RefCell<EventBus>>) -> Self {
        Widget { name: name.to_string(), bus }
    }

    fn click(&self) { // &self: 위젯은 불변
        let event = format!("click:{}", self.name);
        self.bus.borrow().publish(&event);
    }
}

fn main() {
    let bus = Rc::new(RefCell::new(EventBus::new()));

    // 리스너 등록 (bus 공유 소유)
    bus.borrow_mut().subscribe(|e| println!("로그: {}", e));
    bus.borrow_mut().subscribe(|e| println!("분석: {}", e));

    // 두 위젯이 같은 bus 공유
    let btn_a = Widget::new("버튼A", Rc::clone(&bus));
    let btn_b = Widget::new("버튼B", Rc::clone(&bus));

    btn_a.click(); // "로그: click:버튼A", "분석: click:버튼A"
    btn_b.click(); // "로그: click:버튼B", "분석: click:버튼B"
}

10 self / &self / &mut self 공식 문서 →

메서드가 자신을 어떻게 받느냐에 따라 소유권 동작이 완전히 달라집니다. 잘못 선택하면 컴파일 에러가 나거나 불필요하게 값이 소비됩니다.

그림 3. self / &self / &mut self 소유권 동작 비교

실전 예제 — 빌더 패턴(self), 조회(&self), 변경(&mut self)

struct QueryBuilder {
    table: String,
    conditions: Vec<String>,
    limit: Option<usize>,
}

impl QueryBuilder {
    // &self — 읽기만
    fn table_name(&self) -> &str { &self.table }
    fn condition_count(&self) -> usize { self.conditions.len() }

    // &mut self — 상태 변경, 호출 후에도 builder 사용 가능
    fn add_condition(&mut self, cond: &str) {
        self.conditions.push(cond.to_string());
    }
    fn set_limit(&mut self, n: usize) { self.limit = Some(n); }

    // self — 소유권 소비, 최종 결과 생성 (빌더 패턴 끝)
    fn build(self) -> String {
        let where_clause = if self.conditions.is_empty() {
            String::new()
        } else {
            format!(" WHERE {}", self.conditions.join(" AND "))
        };
        let limit_clause = self.limit
            .map(|n| format!(" LIMIT {}", n))
            .unwrap_or_default();
        format!("SELECT * FROM {}{}{}", self.table, where_clause, limit_clause)
    } // self가 여기서 drop됨 — build() 이후 builder 사용 불가
}

fn main() {
    let mut qb = QueryBuilder {
        table: "users".to_string(),
        conditions: vec![],
        limit: None,
    };

    println!("테이블: {}", qb.table_name()); // &self
    qb.add_condition("age > 18");             // &mut self
    qb.add_condition("active = true");          // &mut self
    qb.set_limit(10);                           // &mut self
    println!("조건 수: {}", qb.condition_count()); // &self, 2

    let sql = qb.build(); // self — qb 소비됨
    println!("{}", sql);
    // SELECT * FROM users WHERE age > 18 AND active = true LIMIT 10
    // println!("{}", qb.table_name()); // 에러! qb 이동됨
}

11 for 루프와 이터레이터 공식 문서 →

for x in yyy에서 yyy에 무엇을 쓰느냐가 소유권 동작을 결정합니다. &v, &mut v, v 세 가지는 각각 iter(), iter_mut(), into_iter()를 암묵적으로 호출합니다.

그림 4. iter / iter_mut / into_iter 소유권 비교

fn main() {
    let mut v = vec![10, 20, 30, 40];

    // &v → iter() → x: &i32 — v 유효 유지
    let sum: i32 = v.iter().filter(|&&x| x > 15).sum();
    println!("15 초과 합계: {}, v: {:?}", sum, v); // v 여전히 유효

    // &mut v → iter_mut() → x: &mut i32 — 요소 수정
    for x in &mut v { *x *= 2; }
    println!("두 배: {:?}", v); // [20, 40, 60, 80]

    // v → into_iter() → x: i32 — v 소비
    let big: Vec<i32> = v.into_iter().filter(|&x| x > 40).collect();
    println!("40 초과: {:?}", big); // [60, 80]
    // println!("{:?}", v); // 에러! v는 into_iter()로 소비됨

    // HashMap도 동일 패턴
    use std::collections::HashMap;
    let mut scores: HashMap<String, i32> =
        [("Alice", 95), ("Bob", 87), ("Carol", 92)]
        .iter().map(|(k, v)| (k.to_string(), *v)).collect();

    for (name, score) in &scores {      // &HashMap: (k,v) 참조
        println!("{}: {}", name, score);
    }
    for (_, score) in &mut scores {    // &mut HashMap: 값 수정
        *score += 5;
    }
    println!("{:?}", scores); // 모두 +5
}

12 빌림 규칙과 NLL 공식 문서 →

빌림 검사기(borrow checker)는 컴파일 타임에 메모리 안전성을 보장합니다. 두 가지 핵심 규칙과 Rust 2018+에서 도입된 NLL(Non-Lexical Lifetimes)을 이해하면 컴파일 에러를 예측하고 해결할 수 있습니다.

빌림 규칙 (The Rust Reference, Ch.10.3)
R1. 불변 참조(&T)는 동시에 개수 제한 없이 존재 가능
R2. 가변 참조(&mut T)는 딱 하나만 존재 가능
R3. 가변 참조가 살아있는 동안 불변 참조도 불가 (R1과 R2 동시 불가)

그림 5. 빌림 규칙 스코프 타임라인 — 케이스 1~4

NLL (Non-Lexical Lifetimes) 란?

Rust 2018 이전에는 참조의 수명이 중괄호 블록 끝까지였습니다. Rust 2018+의 NLL은 마지막으로 사용된 시점이 수명 끝입니다. 덕분에 불필요한 블록 중첩 없이 더 자연스러운 코드를 작성할 수 있습니다.

fn main() {
    let mut v = vec![1, 2, 3];

    // NLL 덕분에 가능한 패턴
    let first = &v[0];          // 불변 참조 시작
    println!("first = {}", first); // ← NLL: first의 수명 끝
    v.push(4);                  // OK! first 수명이 이미 끝남
    println!("{:?}", v);        // [1, 2, 3, 4]

    // Rust 2015 스타일 (NLL 이전): 불필요한 블록 필요
    // { let first = &v[0]; println!("{}", first); }
    // v.push(4);

    // 컴파일 에러 케이스 — 에러 메시지 포함
    let mut s = String::from("hello");
    let r1 = &s;
    let r2 = &s;
    println!("{} {}", r1, r2); // r1, r2 수명 끝
    let r3 = &mut s;           // OK! r1, r2 이미 끝남
    println!("{}", r3);
}

13 RefCell 런타임 빌림 검사 — 전체 케이스 공식 문서 →

RefCell은 동일한 빌림 규칙을 컴파일 타임 대신 런타임에 검사합니다. borrow()와 borrow_mut()의 실패 조건, 그리고 패닉 없이 처리하는 try_borrow 방법을 완전히 정리합니다.

use std::cell::RefCell;

fn main() {
    let cell = RefCell::new(vec![1, 2, 3]);

    // ── 케이스 1: 불변 빌림 여러 개 동시 → OK ──
    {
        let r1 = cell.borrow(); // BorrowFlag: 0 → 1
        let r2 = cell.borrow(); // BorrowFlag: 1 → 2
        println!("동시 불변: {:?}, {:?}", *r1, *r2);
    } // r2 drop(2→1), r1 drop(1→0)

    // ── 케이스 2: 가변 빌림 하나 → OK ──
    {
        let mut w = cell.borrow_mut(); // flag: 0 → -1 (가변)
        w.push(4);
        w.retain(|&x| x % 2 == 0); // 짝수만 남김
    } // w drop: flag → 0
    println!("짝수만: {:?}", cell.borrow()); // [2, 4]

    // ── 케이스 3: 불변 중 가변 시도 → PANIC (borrow_mut)
    //    try_borrow_mut으로 안전하게 처리 ──
    let _guard = cell.borrow(); // 불변 빌림 활성
    match cell.try_borrow_mut() {
        Ok(mut w) => w.push(99),
        Err(e) => println!("패닉 없이 처리: {}", e),
        // "already borrowed: BorrowMutError"
    }
    drop(_guard); // 명시적 해제

    // ── 케이스 4: 가변 중 가변 시도 → PANIC ──
    let _w1 = cell.borrow_mut();
    match cell.try_borrow_mut() {
        Ok(_)  => println!("가변 빌림 성공"),
        Err(e) => println!("이미 가변 빌림 중: {}", e),
    }
}

빌림 없음 (flag=0)OK (flag +1)OK (flag=-1)
불변 빌림 N개 (flag>0)OK (flag +1)PANIC: already borrowed
가변 빌림 1개 (flag<0)PANIC: already mutably borrowedPANIC: already mutably borrowed

15 실전 패턴 모음

지금까지 배운 개념들을 실제 코드에서 어떻게 조합하는지 보여주는 완성형 예제들입니다.

패턴 1 — 공유 카운터 (멀티 소유자 + 변경)

use std::rc::Rc;
use std::cell::RefCell;

fn main() {
    let counter = Rc::new(RefCell::new(0));

    let workers: Vec<_> = (0..5).map(|i| {
        let c = Rc::clone(&counter);
        move || {
            *c.borrow_mut() += 1;
            println!("worker {} 완료, count={}", i, c.borrow());
        }
    }).collect();

    for w in workers { w(); }
    println!("최종: {}", counter.borrow()); // 5
}

패턴 2 — 레이지 초기화 (내부 가변성)

use std::cell::RefCell;

struct ExpensiveConfig {
    raw: String,
    parsed: RefCell<Option<Vec<String>>>,
}

impl ExpensiveConfig {
    fn new(raw: &str) -> Self {
        ExpensiveConfig {
            raw: raw.to_string(),
            parsed: RefCell::new(None),
        }
    }

    // &self 이지만 첫 호출 시 캐시를 채움
    fn items(&self) -> Vec<String> {
        if self.parsed.borrow().is_none() {
            // 무거운 파싱 작업 (한 번만 실행)
            let result: Vec<String> = self.raw
                .split(',')
                .map(|s| s.trim().to_string())
                .collect();
            println!("파싱 완료 ({} 항목)", result.len());
            *self.parsed.borrow_mut() = Some(result);
        }
        self.parsed.borrow().clone().unwrap()
    }
}

fn main() {
    let cfg = ExpensiveConfig::new("alpha, beta, gamma, delta");
    println!("{:?}", cfg.items()); // 파싱 완료 (4 항목)
    println!("{:?}", cfg.items()); // 파싱 없이 캐시에서
    println!("첫 번째: {}", cfg.items()[0]);
}

패턴 3 — 방향 그래프 (Rc + RefCell + Weak)

use std::rc::{Rc, Weak};
use std::cell::RefCell;

type NodeRef = Rc<RefCell<GraphNode>>;

struct GraphNode {
    id: usize,
    label: String,
    outgoing: Vec<NodeRef>,           // 강한 참조 (소유)
    incoming: Vec<Weak<RefCell<GraphNode>>>, // 약한 참조 (순환 방지)
}

impl GraphNode {
    fn new(id: usize, label: &str) -> NodeRef {
        Rc::new(RefCell::new(GraphNode {
            id, label: label.to_string(),
            outgoing: vec![], incoming: vec![],
        }))
    }

    fn connect(from: &NodeRef, to: &NodeRef) {
        from.borrow_mut().outgoing.push(Rc::clone(to));
        to.borrow_mut().incoming.push(Rc::downgrade(from));
    }
}

fn main() {
    let a = GraphNode::new(0, "A");
    let b = GraphNode::new(1, "B");
    let c = GraphNode::new(2, "C");

    GraphNode::connect(&a, &b); // A → B
    GraphNode::connect(&b, &c); // B → C
    GraphNode::connect(&a, &c); // A → C

    println!("A의 이웃:");
    for n in &a.borrow().outgoing {
        println!("  → {}", n.borrow().label);
    }
    println!("C로 들어오는 엣지 수: {}",
        c.borrow().incoming.iter()
            .filter(|w| w.upgrade().is_some())
            .count()); // 2 (A, B로부터)
}

타입소유자가변성 검사스레드실패주 용도
Box<T>1명컴파일안전빌드 에러힙 할당, 재귀 타입, dyn Trait
Rc<T>여러 명컴파일단일만빌드 에러공유 그래프·트리 (읽기)
Weak<T>0명(참조만)컴파일단일만upgrade()→None순환 참조 방지
RefCell<T>1명런타임단일만런타임 패닉내부 가변성
Rc<RefCell<T>>여러 명런타임단일만런타임 패닉공유 + 내부 변경
Arc<T>여러 명컴파일안전빌드 에러멀티스레드 공유 (읽기)
