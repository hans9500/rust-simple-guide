Rust 모듈 시스템 — 5가지 핵심 개념

프로젝트가 커질수록 코드를 여러 파일과 모듈로 나눠야 합니다. Rust Book Ch.7은 다음 다섯 가지를 핵심으로 설명합니다. 이 개념들이 서로 어떻게 맞물리는지가 전체 이해의 열쇠입니다.

개념정의 (Rust Book 공식)비유
Package하나 이상의 crate를 포함하는 Cargo 단위. Cargo.toml이 포함됨. rustc는 Package를 모릅니다 — Cargo만 아는 개념회사 전체
CrateRust 컴파일러가 한 번에 처리하는 최소 코드 단위. binary 또는 library 두 종류건물
Module코드를 논리적으로 묶고 가시성(privacy)을 제어하는 단위. mod 키워드로 선언부서
Path모듈 트리 안의 항목을 가리키는 주소. ::으로 구분사내 주소록
use긴 경로를 현재 스코프에 가져와 짧게 쓸 수 있게 하는 키워드단축 다이얼
📌 Rust Book 공식 — 모듈 시스템이 제공하는 기능
  • 코드 조직화: 관련 코드를 모듈로 묶어 읽기 쉽게 만들기
  • privacy 제어: 구현 세부사항을 숨기고 공개 API만 노출하기
  • 이름 충돌 방지: 다른 모듈의 동일한 이름이 충돌하지 않게 하기
  • 재사용성: 잘 정의된 모듈은 다른 프로젝트에서 crate로 사용 가능
섹션 1

Package & Crate

Rust Book은 "crate는 Rust 컴파일러가 한 번에 처리하는 최소 코드 단위"라고 정의합니다. 단일 .rs 파일 하나도 crate입니다.

// 파일 구조: binary + library 공존 (매우 일반적인 패턴)
//
// my_app/
// ├── Cargo.toml
// └── src/
//     ├── main.rs   ← binary crate "my_app" (fn main 포함)
//     └── lib.rs    ← library crate "my_app" (fn main 없음)

// [src/lib.rs] — 핵심 로직을 library에 구현
pub fn format_name(first: &str, last: &str) -> String {
    format!("{} {}", last, first)
}

pub fn sanitize(input: &str) -> String {
    input.trim().to_lowercase()
}

// [src/main.rs] — library를 외부 crate처럼 사용
// Cargo.toml의 name = "my_app" 이므로 crate 이름도 "my_app"
use my_app::format_name;  // 같은 패키지의 lib을 외부 crate처럼 가져옴

fn main() {
    let name = format_name("Gildong", "Hong");
    println!("{}", name); // "Hong Gildong"
}

// Rust Book 권장: 핵심 로직은 lib.rs에 두어
// 다른 프로젝트도 이 library를 사용할 수 있게 하기
// src/bin/ 안의 파일 하나하나가 별도의 binary crate

// 파일 구조:
// my_app/
// ├── Cargo.toml
// └── src/
//     ├── lib.rs              ← library crate
//     ├── main.rs             ← binary crate "my_app"
//     └── bin/
//         ├── server.rs       ← binary crate "server"
//         └── client.rs       ← binary crate "client"

// [src/bin/server.rs]
use my_app::format_name; // lib.rs의 공개 함수 사용

fn main() {
    println!("서버 시작...");
    let name = format_name("Server", "Rust");
    println!("실행: {}", name);
}

// [src/bin/client.rs]
use my_app::sanitize;

fn main() {
    println!("클라이언트 시작...");
    let clean = sanitize("  HELLO  ");
    println!("정리됨: {}", clean);
}

// 실행 방법:
// cargo run              → src/main.rs
// cargo run --bin server → src/bin/server.rs
// cargo run --bin client → src/bin/client.rs
// cargo build --bins     → 모든 binary 빌드
// Cargo.toml 핵심 설정

// [package]
// name    = "my_app"      ← crate 이름 (binary, library 모두 이 이름)
// version = "0.1.0"
// edition = "2021"        ← 2018+ 방식 (extern crate 불필요, mod.rs 선택사항)
//
// [dependencies]
// serde     = { version = "1", features = ["derive"] }
// rand      = "0.8"
// tokio     = { version = "1", features = ["full"] }
//
// # 추가 binary를 명시적으로 등록할 수도 있음
// [[bin]]
// name = "my_tool"
// path = "src/bin/tool.rs"
//
// # 여러 binary 등록
// [[bin]]
// name = "server"
// path = "src/bin/server.rs"
//
// [[bin]]
// name = "client"
// path = "src/bin/client.rs"

// edition = "2021" 이면:
// - mod.rs 없이 파일명으로 모듈 선언 가능
// - extern crate 선언 불필요 (use만으로 충분)
// - Rust 2018 이후 모든 기능 사용 가능
// Rust Book Ch.7 권장 패턴:
// "binary + library 패키지에서 binary crate는
//  library crate를 호출하는 시작점만 담아야 합니다.
//  이렇게 하면 다른 프로젝트도 library의 기능을 사용할 수 있습니다."

// ✅ 좋은 패턴: 로직은 lib.rs에
// [src/lib.rs]
pub mod config {
    pub struct AppConfig {
        pub host: String,
        pub port: u16,
        pub debug: bool,
    }
    impl AppConfig {
        pub fn from_env() -> Self {
            AppConfig {
                host: std::env::var("HOST").unwrap_or("localhost".into()),
                port: std::env::var("PORT")
                    .ok().and_then(|p| p.parse().ok()).unwrap_or(8080),
                debug: std::env::var("DEBUG").is_ok(),
            }
        }
    }
}

pub mod server {
    use super::config::AppConfig;
    pub fn run(cfg: &AppConfig) {
        println!("서버 실행: {}:{}", cfg.host, cfg.port);
    }
}

// [src/main.rs] — 최소한의 코드만
use my_app::{config::AppConfig, server};

fn main() {
    let cfg = AppConfig::from_env();
    server::run(&cfg);
}
// binary crate는 library crate를 "외부 crate처럼" 사용합니다.
// main.rs에서는 pub(crate) 항목에 접근 불가 — 완전히 다른 crate!
섹션 2

mod — 모듈 선언

모듈은 코드를 논리적으로 묶고 가시성을 제어하는 단위입니다. mod 키워드로 선언합니다. Rust Reference는 "모듈은 crate를 구성하는 모듈 트리에 새로운 이름의 모듈을 도입한다"고 정의합니다.

컴파일러의 모듈 파일 탐색 순서 (Rust Reference 공식)

📌 Rust Reference — mod garden; 을 만나면 컴파일러가 찾는 순서
  1. 인라인: mod garden { /* 코드 */ } — 중괄호 안에 바로 코드가 있으면 이것을 사용
  2. 같은 디렉토리의 파일: src/garden.rs
  3. 디렉토리 안의 mod.rs: src/garden/mod.rs (구버전 방식, rustc 1.30 이전 유일한 방법)

⚠️ 같은 모듈에 garden.rs와 garden/mod.rs를 동시에 두면 컴파일 에러입니다!

// 방법 1: 같은 파일 안에 mod 블록으로 정의 (인라인)
// 규모가 작거나 관련성이 높은 코드에 적합

// src/lib.rs
mod front_of_house {        // private: 같은 crate root에서 접근 가능
    pub mod hosting {       // pub: 외부에서 접근 가능
        pub fn add_to_waitlist() {
            println!("대기 목록 추가");
        }
        fn seat_at_table() {    // private: hosting 모듈 안에서만
            println!("좌석 배정");
        }
    }

    mod serving {           // private: front_of_house 안에서만
        fn take_order() {}
        fn serve_order() {}
        fn take_payment() {}
    }
}

// 모듈 트리 (컴파일러가 인식하는 구조):
// crate
//  └── front_of_house
//      ├── hosting
//      │   ├── add_to_waitlist  (pub)
//      │   └── seat_at_table    (private)
//      └── serving              (private)
//          ├── take_order
//          ├── serve_order
//          └── take_payment

pub fn eat_at_restaurant() {
    // front_of_house는 private이지만
    // eat_at_restaurant와 형제(sibling)이므로 접근 가능!
    front_of_house::hosting::add_to_waitlist();
}
// 방법 2: 별도 파일로 분리 (2018+ 권장)
// mod garden; 를 보면 컴파일러가 src/garden.rs 를 찾음

// 파일 구조:
// src/
// ├── lib.rs
// └── garden.rs       ← 이 파일 자체가 garden 모듈의 내용

// [src/lib.rs]
mod garden;             // 세미콜론! → src/garden.rs 를 찾아 garden 모듈로 로드
pub use garden::Vegetable; // garden 내부 항목을 재공개 (pub use)

}

pub fn harvest_season() -> &'static str { "가을" }

// crate::garden::vegetables::Asparagus  ← 절대경로
// crate::garden::flowers::count()       ← 절대경로
// 2015 mod.rs 방식 vs 2018+ 방식 비교
// Rust Reference: "rustc 1.30 이전에는 mod.rs가 유일한 방법이었다"

// ── 2015 스타일 (mod.rs 방식) ─────────────────────────────
// src/
// ├── lib.rs
// └── garden/
//     ├── mod.rs          ← garden 진입점 (항상 mod.rs)
//     ├── vegetables.rs
//     └── flowers.rs
//
// 문제: 에디터에서 여러 탭이 전부 "mod.rs"로 표시 → 어느 모듈인지 구분 어려움
// ─────────────────────────────────────────────────────────────

// ── 2018+ 스타일 (권장) ──────────────────────────────────────
// src/
// ├── lib.rs
// ├── garden.rs            ← garden 진입점 (파일명 = 모듈명 ✅)
// └── garden/
//     ├── vegetables.rs
//     └── flowers.rs
//
// 장점: 파일명으로 모듈 즉시 구분 가능
// ─────────────────────────────────────────────────────────────

// ⚠️ 절대 하면 안 되는 것: 둘을 동시에 사용
// src/garden.rs   AND   src/garden/mod.rs → 컴파일 에러!
// error: file for module `garden` found at both
//        "src/garden.rs" and "src/garden/mod.rs"

// Cargo.toml에 edition = "2021" 이면 2018+ 방식이 기본
// mod.rs 방식은 여전히 지원되지만 권장하지 않음
// #[path] 어트리뷰트 — 파일 위치 직접 지정
// Rust Reference: "path 어트리뷰트로 외부 파일 모듈의
//                  디렉토리와 파일을 직접 지정할 수 있다"

// 기본 탐색 경로와 다른 위치의 파일을 모듈로 로드할 때 사용

// src/lib.rs

// 디렉토리 전체를 다른 이름으로 지정
#[path = "thread_files"]

    #[path = "tls.rs"]
    mod local_data;  // src/thread_files/tls.rs 를 로드
}

// 실전 사용 사례:
// 1. 플랫폼별 구현 분리
#[cfg(target_os = "windows")]
#[path = "platform/windows.rs"]
mod platform;

#[cfg(target_os = "linux")]
#[path = "platform/linux.rs"]
mod platform;

// 2. 레거시 프로젝트의 파일 구조가 Rust 관례와 다를 때
#[path = "../../shared/utils.rs"]
mod shared_utils;

// 주의: #[path]는 상대 경로를 현재 소스 파일 위치 기준으로 해석
// 모듈 중첩 — 모듈 안에 또 모듈
// 모듈은 임의로 깊게 중첩 가능

mod mathematics {
    pub mod algebra {
        pub mod linear {
            pub fn solve_linear(a: f64, b: f64) -> f64 {
                if a == 0.0 { panic!("계수 a는 0이 될 수 없음"); }
                -b / a
            }
            pub fn dot_product(v1: &[f64], v2: &[f64]) -> f64 {
                v1.iter().zip(v2.iter()).map(|(a, b)| a * b).sum()
            }
        }
        pub mod quadratic {
            pub fn discriminant(a: f64, b: f64, c: f64) -> f64 {
                b * b - 4.0 * a * c
            }
            pub fn solve(a: f64, b: f64, c: f64) -> Option<(f64, f64)> {
                let d = discriminant(a, b, c);
                if d < 0.0 { return None; }
                let sqrt_d = d.sqrt();
                Some(((-b + sqrt_d) / (2.0 * a), (-b - sqrt_d) / (2.0 * a)))
            }
        }
    }
    pub mod calculus {
        pub fn derivative(f: impl Fn(f64) -> f64, x: f64) -> f64 {
            let h = 1e-8;
            (f(x + h) - f(x - h)) / (2.0 * h)
        }
    }
}

fn main() {
    // 절대경로
    let x = crate::mathematics::algebra::linear::solve_linear(2.0, -4.0);
    println!("2x - 4 = 0 → x = {}", x); // 2.0

    // 이차방정식 풀기
    if let Some((r1, r2)) = mathematics::algebra::quadratic::solve(1.0, -5.0, 6.0) {
        println!("x² - 5x + 6 = 0 → x = {} or {}", r1, r2); // 3, 2
    }

    // 도함수 (수치 미분)
    let f = |x: f64| x * x; // f(x) = x²
    let df_at_3 = mathematics::calculus::derivative(f, 3.0);
    println!("f'(3) ≈ {} (정확한 값: 6)", df_at_3);
}

🍽️ Rust Book 공식 예제 — 레스토랑

// src/lib.rs (Rust Book Listing 7-1)
// 기본적으로 모든 모듈과 함수는 private
mod front_of_house {
    mod hosting {
        fn add_to_waitlist() {}
        fn seat_at_table() {}
    }
    mod serving {
        fn take_order() {}
        fn serve_order() {}
        fn take_payment() {}
    }
}

// 핵심 privacy 규칙 (Rust Book):
// ① 자식 모듈의 항목은 기본적으로 parent에서 접근 불가 (private)
// ② 자식 모듈은 ancestor의 항목에 접근 가능 (자신이 정의된 맥락을 알기 때문)
// ③ 형제(sibling) 모듈끼리는 private 항목에 접근 불가

// eat_at_restaurant 와 front_of_house 는 형제(sibling) —
// front_of_house 가 private이어도 같은 레벨이라 참조 가능
pub fn eat_at_restaurant() {
    // front_of_house::hosting::add_to_waitlist(); // 에러! hosting이 private
}

// 모듈 트리:
// crate
//  └── front_of_house      (private)
//      ├── hosting          (private)
//      │   ├── add_to_waitlist
//      │   └── seat_at_table
//      └── serving          (private)
//          ├── take_order
//          ├── serve_order
//          └── take_payment
// Rust Book Listing 7-9
// struct: pub으로 선언해도 필드는 기본 private
// 각 필드에 개별적으로 pub 지정 필요

mod back_of_house {
    pub struct Breakfast {
        pub toast: String,           // pub: 외부에서 읽기/쓰기 가능
        seasonal_fruit: String,      // private: 외부에서 직접 접근 불가
    }

    impl Breakfast {
        // private 필드가 있으므로 반드시 공개 생성자 제공
        // (외부에서 Breakfast { ... } 리터럴 생성 불가)
        pub fn summer(toast: &str) -> Breakfast {
            Breakfast {
                toast: String::from(toast),
                seasonal_fruit: String::from("peaches"),  // 내부에서만 설정
            }
        }
        pub fn get_fruit(&self) -> &str {

경로 (Paths) & crate::

경로는 모듈 트리 안의 항목을 가리키는 주소입니다. Rust Book은 두 가지 형태를 정의합니다: 절대경로(crate root에서 시작)와 상대경로(현재 모듈에서 시작).

📌 Rust Book 공식 — 경로의 두 형태
  • 절대경로(Absolute path): crate root부터 시작. 현재 crate 내 코드는 crate 키워드로 시작. 외부 crate는 crate 이름(예: std)으로 시작.
  • 상대경로(Relative path): 현재 모듈에서 시작. self, super, 또는 현재 모듈의 식별자로 시작.
  • 두 형태 모두 ::으로 식별자를 연결.

단계별로 이해하는 경로 — 컴파일 에러부터 해결까지

Rust Book: "super를 사용하면 현재 모듈이 아닌 부모 모듈에서 시작하는 상대경로를 만들 수 있습니다. 파일시스템의 .. 문법과 같습니다."

📌 Rust Book — super:: 를 쓰는 이유

pub use — 재공개 (Re-exporting)

Rust Book: "use 로 이름을 스코프에 가져오면 그 이름은 그 스코프에서 private입니다. pub use를 쓰면 외부 코드도 그 이름을 마치 그 스코프에 정의된 것처럼 사용할 수 있습니다. 이를 재공개(re-exporting)라 합니다."

📌 Rust Book — pub use 가 필요한 이유

pub struct QueryBuilder {
    table: String,           // private: Builder 내부
    conditions: Vec, // private
    limit: Option,      // private
}
impl QueryBuilder {
    pub fn new(table: &str) -> Self {
        QueryBuilder { table: table.into(), conditions: Vec::new(), limit: None }
    }
    pub fn where_clause(mut self, cond: &str) -> Self {
        self.conditions.push(cond.into()); self
    }
    pub fn limit(mut self, n: u32) -> Self {
        self.limit = Some(n); self
    }
    pub fn build(&self) -> String {
        let mut sql = format!("SELECT * FROM {}", self.table);
        if !self.conditions.is_empty() {
            sql += &format!(" WHERE {}", self.conditions.join(" AND "));
        }
        if let Some(l) = self.limit { sql += &format!(" LIMIT {}", l); }
        sql
    }
}

pub fn slugify(s: &str) -> String {
    s.to_lowercase()
        .chars()
        .map(|c| if c.is_alphanumeric() { c } else { '-' })

        .trim_matches('-')
        .to_string()

✅ Rust Book 최종 정리 — 모듈 시스템의 핵심 원칙
  • 기본은 private: 모든 항목은 기본적으로 private. pub을 명시해야 공개
  • 자식은 부모를 알지만, 부모는 자식을 모른다: 자식 모듈은 ancestor의 private에 접근 가능
  • 모듈 공개 ≠ 내용 공개: pub mod hosting이어도 내부 함수는 별도로 pub 필요
  • struct와 enum의 pub 차이: struct는 필드별 지정, enum은 전체 자동 공개
  • use는 스코프에 귀속: 선언된 모듈/함수/블록 스코프에서만 유효
  • 절대경로 권장: Rust Book은 코드 이동 안정성을 위해 crate:: 절대경로를 선호
  • pub use로 API 설계: 내부 구조와 공개 API를 분리해 사용자 편의 제공
