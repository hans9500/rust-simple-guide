01

mut의 진짜 의미 — 바인딩과 접근 방식의 속성

핵심 질문

왜 Rust는 Java처럼 필드별로 final을 붙이지 않을까? mut은 정확히 무엇에 붙는 것인가?

Rust 공식 문서는 mutability(가변성)는 바인딩(binding) 또는 빌림(&mut)의 속성이지, 값(value) 자체의 속성이 아니라고 명확히 정의합니다. 이것은 Java의 final이나 C++의 const와 근본적으로 다릅니다.

Mutability is a property of either a borrow (&mut) or a binding (let mut). This means that, for example, you cannot have a struct with some fields mutable and some immutable. — The Rust Programming Language (공식 Book, Mutability 챕터)

또한 Rust Internals 토론에서 코어 팀이 명확히 밝힌 중요한 사실이 있습니다: "Rust에는 진짜 immutable 데이터가 존재하지 않습니다." let으로 선언한 변수가 불변처럼 보이는 것은 단지 그 바인딩을 통한 접근이 제한되기 때문입니다. 소유권을 이동(move)하면 그 제한이 사라집니다.

EX 01 소유권 이동 후 가변성 변화 — let이 불변을 보장하는 것이 아님을 증명
ownership_mutability.rs RUST 
fn main() {
    // ── 1. immutable 바인딩 ──
    let s1 = String::from("hello");
    // s1.push_str(" world");  ← 컴파일 오류 E0596

    // ── 2. 소유권 이동 후 mutable 바인딩으로 받기 ──
    // s1은 immutable이었지만, 소유권을 이동하면서 mut로 받을 수 있음
    let mut s2 = s1;   // s1의 소유권이 s2로 이동
    s2.push_str(" world");   // 이제 수정 가능!
    println!("{}", s2);   // hello world

    // ── 3. 반대 방향: mut에서 immutable로 ──
    let mut v = vec![1, 2, 3];
    v.push(4);

    let v_frozen = v;   // immutable 바인딩으로 이동
    // v_frozen.push(5);  ← 이제 오류 — 바인딩이 immutable
    println!("{:?}", v_frozen);   // [1, 2, 3, 4]

    // ── 4. 함수에 mut로 넘기기 ──
    let data = vec![10, 20, 30];
    process(data);   // immutable 바인딩이었지만 함수 안에서 mut로 받음
}

fn process(mut v: Vec<i32>) {
    // 매개변수에 mut를 붙이면 이 함수 내에서는 수정 가능
    v.push(99);
    println!("함수 내부: {:?}", v);   // [10, 20, 30, 99]
}   // v는 여기서 drop됨 — 호출자의 data와 무관
Java 개발자를 위한 비교

Java의 final은 참조 자체를 고정합니다. Rust의 let은 이보다 넓은 개념으로, 소유권 이동이 발생하면 가변성 제약이 재설정됩니다. Java에는 이에 대응하는 개념이 없습니다.

EX 02 shadowing과 mut의 차이 — 완전히 다른 메커니즘
shadowing_vs_mut.rs RUST 
fn main() {
    // ── shadowing: let을 다시 써서 새 변수 생성 ──
    let spaces = "   ";           // &str 타입
    let spaces = spaces.len();   // usize 타입 — 타입이 바뀜!
    println!("spaces = {}", spaces);   // 3

    // ── mut: 같은 변수에 같은 타입으로 재할당 ──
    let mut count = 0;
    count += 1;
    count += 1;
    println!("count = {}", count);   // 2

    // ── 차이 1: shadowing은 타입을 바꿀 수 있다 ──
    let x = 5;
    let x = x * 2;            // 새 변수 (i32)
    let x = format!("{}", x);  // 새 변수 (String) — 타입 변환!
    println!("x = {}", x);      // "10"

    // ── 차이 2: mut는 타입을 바꿀 수 없다 ──
    let mut y = 5;
    y = 10;   // OK — 같은 타입
    // y = "hello";  ← 오류 — 타입 불일치

    // ── 차이 3: shadowing은 스코프를 존중한다 ──
    let z = 1;
    {
        let z = z + 10;   // 내부 스코프 z = 11
        println!("inner z = {}", z);   // 11
    }
    println!("outer z = {}", z);   // 1 — 외부 z는 그대로
}
02

상속적 가변성(Inherited Mutability) — Rust의 기본 가변성 모델

상속적 가변성(Inherited Mutability)은 값의 가변성이 소유자 또는 독점 참조(&mut)로부터 상속되는 Rust의 기본 동작입니다. std::cell 공식 문서는 이를 "inherited mutability (mutable only via &mut T)"로 명명하며, Cell/RefCell의 내부 가변성과 대비시킵니다.

The more common inherited mutability, where one must have unique access to mutate a value, is one of the key language elements that enables Rust to reason strongly about pointer aliasing, statically preventing crash bugs. Because of that, inherited mutability is preferred, and interior mutability is something of a last resort. — std::cell 공식 문서

상속적 가변성의 규칙은 단순합니다:

소유권 보유
(let mut x)
→
내부 값 전체 수정 가능
독점 참조
(&mut x)
→
참조 통해 수정 가능
공유 참조
(&x)
→
수정 불가 (기본)
→
Cell/RefCell로만 예외 가능
EX 03 상속적 가변성 — 참조를 통한 수정 패턴 전체
inherited_mutability.rs RUST 
fn main() {
    // ── 패턴 1: 소유권으로 수정 ──
    let mut x = 5;
    x = 10;   // 소유권이 있으니 자유롭게 수정

    // ── 패턴 2: &mut 참조 ──
    let mut y = 5;
    let ry = &mut y;   // 독점 가변 참조
    *ry = 20;           // 역참조로 수정
    // y = 30;  ← 오류: ry가 살아있는 동안 y 직접 접근 불가
    drop(ry);
    println!("y = {}", y);   // 20

    // ── 패턴 3: immutable 변수 = let mut y에서 &만 빌림 ──
    let mut z = 5;
    let rz = &z;   // 불변 참조 (immutable binding to mutable variable)
    println!("rz = {}", *rz);   // 읽기만 가능
    // *rz = 10;  ← 오류: &T를 통해서는 수정 불가
    drop(rz);
    z = 99;   // 불변 참조 해제 후 소유권으로 직접 수정 가능

    // ── 패턴 4: 함수에 &mut 넘기기 ──
    let mut v = vec![1, 2, 3];
    add_element(&mut v);
    println!("{:?}", v);   // [1, 2, 3, 99]

    // ── 핵심: 동시에 &mut는 단 하나만 존재 가능 ──
    let mut a = 0;
    let r1 = &mut a;
    // let r2 = &mut a;  ← 오류: cannot borrow as mutable more than once
    *r1 = 1;
}

fn add_element(v: &mut Vec<i32>) {
    v.push(99);
}
03

구조체와 mut — 필드별 가변성은 없다

구조체 정의에서 필드별로 mut을 붙이는 것은 Rust에서 허용되지 않습니다. 가변성은 값에 어떻게 접근하느냐에 달려 있습니다. let mut p으로 바인딩하면 모든 필드가 수정 가능하고, let p로 바인딩하면 모든 필드가 수정 불가합니다.

이것은 불가능합니다 RUST — 컴파일 오류 
struct Point {
    x: i32,
    mut y: i32,   // ← 문법 오류: 필드에 mut 불가
}
error: expected identifier, found keyword `mut` --> src/main.rs:3:5 | 3 | mut y: i32, | ^^^ expected identifier

단, Cell<T>를 이용하면 필드 단위 가변성을 에뮬레이션할 수 있습니다. 이것이 Rust 공식 Book이 소개하는 패턴입니다.

EX 04 구조체 전체/부분 수정 + Cell로 필드 단위 가변성 에뮬레이션
struct_mutability.rs RUST 
use std::cell::Cell;

// ── 일반 구조체: 바인딩 전체가 기준 ──
struct Point {
    x: i32,
    y: i32,
}

// ── Cell 활용: 필드 단위 가변성 에뮬레이션 ──
// 공식 Book이 소개하는 패턴 (Mutability 챕터)
struct PointHybrid {
    x: i32,           // 일반 필드 — let mut이어야 수정 가능
    y: Cell<i32>,    // Cell 필드 — let이어도 수정 가능!
}

fn main() {
    // ── 케이스 A: immutable 바인딩 — 아무 필드도 수정 불가 ──
    let p1 = Point { x: 1, y: 2 };
    println!("p1 = ({}, {})", p1.x, p1.y);
    // p1.x = 10;  ← E0594: cannot assign to `p1.x` which is behind an immutable binding
    // p1.y = 20;  ← 동일 오류

    // ── 케이스 B: mutable 바인딩 — 모든 필드 수정 가능 ──
    let mut p2 = Point { x: 1, y: 2 };
    p2.x = 10;   // 부분 수정 OK
    p2.y = 20;   // 부분 수정 OK
    println!("p2 = ({}, {})", p2.x, p2.y);

    // 구조체 리터럴로 전체 교체
    p2 = Point { x: 0, y: 0 };
    println!("p2 after replace = ({}, {})", p2.x, p2.y);

    // ── 케이스 C: Cell 필드 — immutable 바인딩에서 y만 수정 ──
    let ph = PointHybrid {
        x: 5,
        y: Cell::new(6),
    };
    // ph.x = 99;  ← 오류 — x는 일반 필드
    ph.y.set(7);   // OK — Cell 필드는 &self로도 수정 가능
    println!("ph.y = {:?}", ph.y);   // Cell { value: 7 }

    // ── 케이스 D: 구조체 Update Syntax ──
    let mut p3 = Point { x: 1, y: 2 };
    p3 = Point { x: 99, ..p3 };   // x만 교체, y는 유지 (Update Syntax)
    println!("p3 = ({}, {})", p3.x, p3.y);   // (99, 2)

    // ── 케이스 E: 구조체 분해(destructure) 후 수정 ──
    let Point { mut x, y } = Point { x: 3, y: 4 };
    x *= 2;   // x만 mut로 분해했으므로 수정 가능
    // y *= 2;  ← 오류 — y는 mut 없이 분해
    println!("x={}, y={}", x, y);   // 6, 4
}
04

&self vs &mut self vs self — 메서드 수신자(Receiver) 완전 정리

메서드의 첫 번째 매개변수(수신자, receiver)는 세 가지 형태가 있으며, 각각 다른 가변성과 소유권 의미론을 가집니다.

수신자 의미 호출 후 소유권 호출 조건 사용 예
&self 불변 공유 참조 호출자가 유지 let / let mut 모두 가능 읽기, 계산, 출력
&mut self 독점 가변 참조 호출자가 유지 let mut 필수 내부 상태 변경
self 소유권 이동 메서드가 소비 let / let mut 모두 가능 소비 후 변환, Builder 패턴
mut self 소유권 이동 + 내부 수정 메서드가 소비 let / let mut 모두 가능 소비하면서 내부 변경
EX 05 네 가지 수신자 — Builder 패턴을 통한 완전한 예제
method_receivers.rs RUST 
#[derive(Debug)]
struct Config {
    host: String,
    port: u16,
    max_conn: usize,
    debug: bool,
}

impl Config {
    // ── 1. &self — 읽기 전용, 호출자가 소유권 유지 ──
    fn address(&self) -> String {
        format!("{}:{}", self.host, self.port)
    }

    fn is_valid(&self) -> bool {
        !self.host.is_empty() && self.port > 0
    }

    // ── 2. &mut self — 내부 수정, 호출자가 소유권 유지 ──
    // 주의: 호출하려면 바인딩이 let mut이어야 함
    fn set_port(&mut self, port: u16) {
        self.port = port;
    }

    fn enable_debug(&mut self) {
        self.debug = true;
    }

    // ── 3. self — 소유권 이동, 호출 후 원본 사용 불가 ──
    fn into_address(self) -> String {
        // self를 소비하면서 값을 반환
        format!("{}:{}", self.host, self.port)
        // 함수 끝에서 self의 나머지 필드가 drop됨
    }

    // ── 4. mut self — 소유권 이동 + 내부 수정 (Builder 패턴) ──
    // 호출자의 바인딩이 immutable이어도 됨 — 소유권이 이동하므로
    fn with_port(mut self, port: u16) -> Self {
        self.port = port;
        self
    }

    fn with_max_conn(mut self, n: usize) -> Self {
        self.max_conn = n;
        self
    }

    fn with_debug(mut self) -> Self {
        self.debug = true;
        self
    }
}

fn main() {
    // ── &self: let 바인딩에서도 호출 가능 ──
    let cfg = Config {
        host: "localhost".to_string(),
        port: 8080,
        max_conn: 100,
        debug: false,
    };
    println!("address = {}", cfg.address());   // &self → let ok
    println!("valid = {}", cfg.is_valid());     // &self → let ok

    // ── &mut self: let mut 필수 ──
    let mut cfg2 = Config {
        host: "0.0.0.0".to_string(),
        port: 80,
        max_conn: 50,
        debug: false,
    };
    cfg2.set_port(443);     // let mut이므로 OK
    cfg2.enable_debug();    // let mut이므로 OK
    println!("cfg2 = {:?}", cfg2);

    // ── Builder 패턴 (mut self) — 체이닝 ──
    // let 바인딩이어도 됨: 소유권이 이동하므로 mut 제약 재설정됨
    let final_cfg = Config {
        host: "api.example.com".to_string(),
        port: 80,
        max_conn: 10,
        debug: false,
    }
    .with_port(443)
    .with_max_conn(1000)
    .with_debug();
    println!("final_cfg = {:?}", final_cfg);

    // ── self: 소유권 소비 ──
    let addr = final_cfg.into_address();
    println!("addr = {}", addr);
    // println!("{:?}", final_cfg);  ← 오류: value moved
}
05

std::mem — replace / swap / take

std::mem 모듈은 소유권 이동 없이 값을 교환하거나 꺼낼 수 있는 함수들을 제공합니다. 이 함수들은 모두 &mut T 참조를 받아, 빈 위치 없이 원자적으로 교환하므로 소유권 안전성이 보장됩니다.

함수시그니처동작반환값
mem::replace (&mut T, T) → T dest에 src 저장, dest의 이전 값 반환 이전 값 (T)
mem::swap (&mut T, &mut T) 두 값의 내용을 서로 교환 ()
mem::take (&mut T) → T 값을 꺼내고 Default::default()로 채움 원래 값 (T)
왜 단순 대입이 아닌가?

self.value = new_value만 하면 기존 값이 drop됩니다. 기존 값의 소유권을 돌려받고 싶다면 replace를 써야 합니다. 또한 &mut self 메서드에서 self.field를 직접 move out하면 컴파일 오류가 나는데, replace/take가 이 문제를 해결합니다.

EX 06 replace / swap / take — 실용 예제
mem_ops.rs RUST 
use std::mem;

#[derive(Debug)]
struct Cache {
    data: Vec<String>,
    dirty: bool,
}

impl Cache {
    fn new() -> Self {
        Cache { data: Vec::new(), dirty: false }
    }

    // mem::replace: 현재 data를 꺼내고 새 빈 Vec으로 교체
    // &mut self에서 self.data를 직접 move-out 하면 컴파일 오류 →
    // replace가 해결책
    fn flush(&mut self) -> Vec<String> {
        self.dirty = false;
        mem::replace(&mut self.data, Vec::new())
        // 또는 더 관용적으로: mem::take(&mut self.data)
    }

    fn push(&mut self, s: &str) {
        self.data.push(s.to_string());
        self.dirty = true;
    }
}

#[derive(Debug)]
struct MyBox<T> {
    value: T,
    label: String,
}

impl<T: std::fmt::Debug> MyBox<T> {
    fn new(value: T, label: &str) -> Self {
        MyBox { value, label: label.to_string() }
    }

    fn get(&self) -> &T { &self.value }

    // replace: 새 값 저장, 이전 값 반환
    fn replace(&mut self, new_value: T) -> T {
        mem::replace(&mut self.value, new_value)
    }

    // take: Default로 채우고 이전 값 반환 (T: Default 필요)
    fn take_label(&mut self) -> String {
        mem::take(&mut self.label)   // label이 ""로 교체되고 이전 값 반환
    }
}

fn main() {
    // ── replace 예제 ──
    let mut b = MyBox::new("hello".to_string(), "greeting");
    println!("before: {:?}", b.get());
    let old = b.replace("world".to_string());
    println!("old = {:?}, now = {:?}", old, b.get());

    // ── take 예제 ──
    let label = b.take_label();
    println!("label = {:?}, b.label = {:?}", label, b.label);
    // "greeting", "" — label이 Default("")로 채워짐

    // ── swap 예제 ──
    let mut x = 100;
    let mut y = 200;
    mem::swap(&mut x, &mut y);
    println!("x={}, y={}", x, y);   // 200, 100

    // ── Cache flush 예제 ──
    let mut cache = Cache::new();
    cache.push("item1");
    cache.push("item2");
    cache.push("item3");
    println!("cache before flush: {:?}", cache);

    let flushed = cache.flush();
    println!("flushed = {:?}", flushed);   // ["item1", "item2", "item3"]
    println!("cache after flush: {:?}", cache);  // 빈 Vec
}
06

내부 가변성(Interior Mutability) — 원리와 UnsafeCell

핵심 질문

왜 let으로 선언한 변수의 내부를 수정할 수 있는가? 컴파일러를 "속이는" 것인가?

내부 가변성(Interior Mutability)이란 &T(공유 참조, shared reference)를 통해서도 내부 값을 변경할 수 있는 패턴입니다. Rust 공식 문서는 이것을 상속적 가변성의 "최후의 수단(last resort)"으로 표현하며, 단일 스레드 환경에서는 Cell/RefCell/OnceCell이, 멀티 스레드 환경에서는 Mutex/RwLock/OnceLock이 이를 담당합니다.

Values of the Cell<T>, RefCell<T>, and OnceCell<T> types may be mutated through shared references (i.e. the common &T type), whereas most Rust types can only be mutated through unique (&mut T) references. We say these cell types provide 'interior mutability' (mutable via &T), in contrast with typical Rust types that exhibit 'inherited mutability' (mutable only via &mut T). — std::cell 공식 문서

내부 가변성이 안전한 이유 — UnsafeCell<T>

컴파일러는 &T가 가리키는 데이터는 불변이라고 가정하고 최적화합니다. 이 가정을 깨는 것은 undefined behavior입니다. 단, UnsafeCell<T>는 이 보장을 명시적으로 opt-out하는 유일한 합법적 수단입니다.

UnsafeCell<T> opts-out of the immutability guarantee for &T: a shared reference &UnsafeCell<T> may point to data that is being mutated. This is called "interior mutability". All other types that allow internal mutability, such as Cell<T> and RefCell<T>, internally use UnsafeCell to wrap their data. — std::cell::UnsafeCell 공식 문서

따라서 Cell/RefCell은 컴파일러를 속이는 것이 아니라, UnsafeCell이라는 공식 탈출구를 통해 안전하게 내부 가변성을 제공합니다. 빌림 규칙은 여전히 적용되되, 검사 시점이 컴파일 타임에서 런타임으로 이동합니다.

공식 문서가 내부 가변성을 권장하는 상황은 다음과 같습니다:

  • 불변처럼 보이는 값의 내부에 가변 상태를 도입할 때 (예: 캐시, 방문 카운터)
  • 논리적으로는 불변인 메서드(&self)의 구현 세부사항으로 내부를 변경할 때
  • Clone 등 트레이트가 &self만 요구하는데 내부 참조 카운트를 증가시킬 때
  • Rc<T>/Arc<T> 같은 공유 스마트 포인터 — multiply-aliased이므로 &mut 불가
07

RefCell<T> — 런타임 빌림 검사기

RefCell<T>는 단일 소유권을 가지며 빌림 규칙을 런타임에 검사합니다. Box<T>와 달리 borrow()/borrow_mut()을 통해 내부 값의 참조를 얻을 수 있고, 내부적으로 현재 활성 빌림 수를 추적합니다.

RefCell<T> uses Rust's lifetimes to implement "dynamic borrowing", a process whereby one can claim temporary, exclusive, mutable access to the inner value. Borrows for RefCell<T>s are tracked at runtime, unlike Rust's native reference types which are entirely tracked statically, at compile time. — std::cell 공식 문서
메서드반환 타입내부 카운터 변화충돌 시
.borrow() Ref<T> 불변 카운트 +1 panic!
.borrow_mut() RefMut<T> 가변 플래그 set panic!
.try_borrow() Result<Ref<T>, BorrowError> 불변 카운트 +1 Err 반환
.try_borrow_mut() Result<RefMut<T>, BorrowMutError> 가변 플래그 set Err 반환
.into_inner() T — 빌림 중이면 panic

refcell_mock.rs RUST 
// 공식 Book Chapter 15.5의 핵심 예제 패턴
// &self만 받는 trait 메서드에서 내부 상태를 변경해야 할 때
use std::cell::RefCell;

trait Messenger {
    fn send(&self, msg: &str);  // &self — &mut self로 바꿀 수 없는 상황
}

// 실제 구현: 메시지를 외부 시스템으로 전송
struct EmailSender;
impl Messenger for EmailSender {
    fn send(&self, msg: &str) {
        println!("[EMAIL] {}", msg);
    }
}

// 테스트용 Mock: 메시지를 내부에 저장
// &self만 받는 send()에서 내부 Vec을 수정해야 → RefCell!
struct MockMessenger {
    sent_messages: RefCell<Vec<String>>,
}

impl MockMessenger {
    fn new() -> Self {
        MockMessenger { sent_messages: RefCell::new(vec![]) }
    }
}

impl Messenger for MockMessenger {
    fn send(&self, msg: &str) {
        // &self인데 내부를 수정! RefCell이 없으면 불가능
        self.sent_messages.borrow_mut().push(msg.to_string());
    }
}

struct LimitChecker<'a, T: Messenger> {
    messenger: &'a T,
    value: usize,
    max: usize,
}

impl<'a, T: Messenger> LimitChecker<'a, T> {
    fn new(messenger: &'a T, max: usize) -> Self {
        LimitChecker { messenger, value: 0, max }
    }

    fn set_value(&mut self, value: usize) {
        self.value = value;
        let pct = self.value as f64 / self.max as f64;
        if pct >= 1.0 {
            self.messenger.send("[ALERT] 한도 초과!");
        } else if pct >= 0.9 {
            self.messenger.send("[WARN] 90% 도달");
        } else if pct >= 0.75 {
            self.messenger.send("[INFO] 75% 도달");
        }
    }
}

fn main() {
    let mock = MockMessenger::new();
    let mut checker = LimitChecker::new(&mock, 100);

    checker.set_value(80);  // 75% 초과 → 경고 발송
    checker.set_value(95);  // 90% 초과 → 경고 발송
    checker.set_value(110); // 100% 초과 → 알람 발송

    // mock은 let (immutable)인데 내부에 3개 메시지가 저장됨
    let messages = mock.sent_messages.borrow();
    assert_eq!(messages.len(), 3);
    println!("발송된 메시지: {:?}", *messages);
}
EX 08 RefCell 패닉 시나리오 — 두 개의 borrow_mut 동시 활성
refcell_panic.rs RUST 
use std::cell::RefCell;

struct MockMessenger {
    sent: RefCell<Vec<String>>,
}

impl MockMessenger {
    // 이 예제는 공식 Book이 런타임 패닉을 설명하는 코드
    fn bad_send(&self, msg: &str) {
        let mut one = self.sent.borrow_mut(); // 첫 번째 가변 빌림
        let mut two = self.sent.borrow_mut(); // 두 번째 — 런타임 panic!
        // 컴파일은 되지만 실행 시 패닉 발생
        one.push(msg.to_string());
        two.push(msg.to_string());
    }
}

fn main() {
    let mock = MockMessenger { sent: RefCell::new(vec![]) };

    // catch_unwind으로 패닉을 잡아서 프로그램이 죽지 않게 함
    let result = std::panic::catch_unwind(std::panic::AssertUnwindSafe(|| {
        mock.bad_send("hello");
    }));

    match result {
        Ok(_) => println!("성공"),
        Err(e) => {
            if let Some(s) = e.downcast_ref::<&str>() {
                println!("패닉 발생: {}", s);
                // "already mutably borrowed"
            }
        }
    }
}

try_borrow / try_borrow_mut — 패닉 없는 안전한 빌림

borrow()/borrow_mut()은 규칙 위반 시 즉시 패닉합니다. 운영 환경에서 패닉은 프로세스 종료를 의미하므로, try_borrow()/try_borrow_mut()으로 Result를 받아 처리하는 것이 권장됩니다.

권장 패턴

운영 코드에서는 borrow_mut() 대신 try_borrow_mut()을 쓰고, Err가 나올 경우 로그를 남기거나 재시도 큐에 넣는 방식으로 설계합니다.

EX 09 try_borrow / try_borrow_mut — 모든 시나리오
try_borrow_all.rs RUST 
use std::cell::RefCell;

fn main() {
    let cell = RefCell::new(100i32);

    // ══ 시나리오 1: 유휴 상태 → 모든 빌림 성공 ══
    println!("=== 시나리오 1: 유휴 상태 ===");
    match cell.try_borrow() {
        Ok(v) => println!("불변 빌림 성공: {}", *v),
        Err(e) => println!("실패: {}", e),
    }
    match cell.try_borrow_mut() {
        Ok(mut v) => {
            *v += 1;
            println!("가변 빌림 성공: {}", *v);
        }
        Err(e) => println!("실패: {}", e),
    }

    // ══ 시나리오 2: 불변 빌림 중 → 가변 빌림 실패 ══
    println!("\n=== 시나리오 2: 불변 빌림 활성 중 ===");
    {
        let r1 = cell.borrow();   // 불변 카운터 = 1
        let r2 = cell.borrow();   // 불변 카운터 = 2 → OK
        println!("r1={}, r2={}", *r1, *r2);

        match cell.try_borrow_mut() {
            Ok(_) => println!("이건 출력 안됨"),
            Err(e) => println!("가변 빌림 실패 (예상): {}", e),
        }
        // r1, r2: 스코프 끝에서 자동 drop → 카운터 = 0
    }

    // ══ 시나리오 3: 가변 빌림 중 → 불변/가변 모두 실패 ══
    println!("\n=== 시나리오 3: 가변 빌림 활성 중 ===");
    {
        let mut w = cell.borrow_mut();   // 가변 플래그 ON
        *w = 999;

        match cell.try_borrow() {
            Ok(_)  => println!("이건 출력 안됨"),
            Err(e) => println!("불변도 실패: {}", e),
        }
        match cell.try_borrow_mut() {
            Ok(_)  => println!("이건 출력 안됨"),
            Err(e) => println!("가변도 실패: {}", e),
        }
    }   // w drop → 가변 플래그 OFF

    // ══ 시나리오 4: drop 후 재시도 → 성공 ══
    println!("\n=== 시나리오 4: drop 후 재시도 ===");
    println!("최종값: {}", cell.borrow());   // 999
}
EX 10 try_borrow_mut 실전 — HTTP 라우터 (재시도 큐 포함)
try_borrow_router.rs RUST 
use std::cell::{Cell, RefCell};
use std::collections::HashMap;

struct Router {
    // 단순 숫자 → Cell (Copy, zero-cost)
    req_count:  Cell<u64>,
    err_count:  Cell<u64>,
    // 복잡한 데이터 → RefCell
    routes:     RefCell<HashMap<String, String>>,
    log:        RefCell<Vec<String>>,
    retry_queue: RefCell<Vec<String>>,
}

impl Router {
    fn new() -> Self {
        Router {
            req_count:   Cell::new(0),
            err_count:   Cell::new(0),
            routes:      RefCell::new(HashMap::new()),
            log:         RefCell::new(Vec::new()),
            retry_queue: RefCell::new(Vec::new()),
        }
    }

    fn register(&self, path: &str, handler: &str) {
        match self.routes.try_borrow_mut() {
            Ok(mut r) => {
                r.insert(path.to_string(), handler.to_string());
                self.append_log(format!("등록: {}", path));
            }
            Err(e) => {
                self.err_count.update(|c| c + 1);
                eprintln!("라우트 등록 실패 ({}): {}", path, e);
            }
        }
    }

    fn handle(&self, path: &str) -> String {
        self.req_count.update(|c| c + 1);

        match self.routes.try_borrow() {
            Ok(r) => {
                let resp = r.get(path)
                    .map(|h| format!("200 OK ({})", h))
                    .unwrap_or_else(|| "404 Not Found".to_string());
                self.append_log(format!("#{} {} → {}",
                    self.req_count.get(), path, resp));
                resp
            }
            Err(e) => {
                // 라우트 테이블 잠김 → 재시도 큐에 등록
                self.err_count.update(|c| c + 1);
                if let Ok(mut q) = self.retry_queue.try_borrow_mut() {
                    q.push(path.to_string());
                }
                format!("503 Service Unavailable ({})", e)
            }
        }
    }

    fn append_log(&self, msg: String) {
        if let Ok(mut l) = self.log.try_borrow_mut() {
            l.push(msg);
        }
    }

    fn stats(&self) {
        println!("=== 통계 ===");
        println!("요청: {}, 오류: {}",
            self.req_count.get(), self.err_count.get());
        let q = self.retry_queue.borrow();
        if !q.is_empty() {
            println!("재시도 대기: {:?}", *q);
        }
        println!("로그 ({} 건):", self.log.borrow().len());
        for entry in self.log.borrow().iter() {
            println!("  {}", entry);
        }
    }
}

fn main() {
    let router = Router::new(); // let (mut 없음!)

    router.register("/",       "index");
    router.register("/api",   "api_handler");
    router.register("/health", "health_check");

    println!("{}", router.handle("/"));
    println!("{}", router.handle("/api"));
    println!("{}", router.handle("/unknown"));

    router.stats();
}

EX 11 Cell — 공식 Book 패턴 + 실전 활용 (방문 카운터, 캐시 플래그)
cell_patterns.rs RUST 
use std::cell::Cell;

// ── 패턴 1: 공식 Book — Cell로 필드 단위 가변성 에뮬레이션 ──
struct Point {
    x: i32,           // immutable 바인딩에서 수정 불가
    y: Cell<i32>,    // immutable 바인딩에서도 수정 가능!
}

// ── 패턴 2: 방문 카운터 (논리적으로 불변인 메서드의 내부 상태) ──
struct Article {
    title: String,
    content: String,
    view_count: Cell<u64>,   // &self로 카운트 증가
    is_cached: Cell<bool>,   // 캐시 상태 플래그
}

impl Article {
    fn new(title: &str, content: &str) -> Self {
        Article {
            title: title.to_string(),
            content: content.to_string(),
            view_count: Cell::new(0),
            is_cached: Cell::new(false),
        }
    }

    // &self: 논리적으로는 "읽기"지만 카운터는 증가해야 함
    // RefCell 없이 Cell만으로 가능 (Copy 타입이므로)
    fn read(&self) -> &str {
        self.view_count.update(|c| c + 1);   // zero-cost!
        if !self.is_cached.get() {
            self.is_cached.set(true);
        }
        &self.content
    }

    fn views(&self) -> u64 { self.view_count.get() }
    fn cached(&self) -> bool { self.is_cached.get() }
}

// ── 패턴 3: Cell::replace와 Cell::take ──
struct StateMachine {
    state: Cell<u8>,
    prev_state: Cell<u8>,
}

impl StateMachine {
    fn new() -> Self {
        StateMachine { state: Cell::new(0), prev_state: Cell::new(0) }
    }

    fn transition(&self, new_state: u8) {
        // replace: 이전 상태를 prev_state에 저장하면서 새 상태로 전환
        let old = self.state.replace(new_state);
        self.prev_state.set(old);
        println!("상태 전환: {} → {}", old, new_state);
    }

    fn rollback(&self) {
        let prev = self.prev_state.get();
        self.state.set(prev);
        println!("롤백: → {}", prev);
    }
}

fn main() {
    // ── Cell 필드 단위 가변성 ──
    let p = Point { x: 5, y: Cell::new(6) };
    // p.x = 10;  ← 컴파일 오류 — 일반 필드
    p.y.set(7);   // OK — Cell 필드
    println!("Point: x={}, y={:?}", p.x, p.y);

    // ── 방문 카운터 ──
    let article = Article::new("Rust Guide", "Rust is awesome!");
    println!("{}", article.read());
    println!("{}", article.read());
    println!("{}", article.read());
    println!("조회수: {}, 캐시됨: {}",
        article.views(), article.cached());   // 3, true

    // ── 상태 머신 ──
    let sm = StateMachine::new();
    sm.transition(1);   // 0 → 1
    sm.transition(2);   // 1 → 2
    sm.rollback();      // → 1
    println!("현재 상태: {}", sm.state.get());   // 1
}

Ref<T> / RefMut<T> — 스마트 포인터와 생명주기

borrow()/borrow_mut()의 반환값은 단순한 참조(&T/&mut T)가 아니라 Ref<T>/RefMut<T> 스마트 포인터입니다. 이 값들은 Drop 트레이트를 구현하여, 스코프를 벗어나거나 drop()이 호출될 때 RefCell의 내부 카운터를 자동으로 감소시킵니다.

The borrow method returns the smart pointer type Ref<T>, and borrow_mut returns the smart pointer type RefMut<T>. Both types implement Deref, so we can treat them like regular references. The RefCell<T> keeps track of how many Ref<T> and RefMut<T> smart pointers are currently active. — The Rust Programming Language (공식 Book, Chapter 15.5)
가장 흔한 패닉 패턴

Ref<T>/RefMut<T>를 변수에 담아두면 해당 스코프가 끝날 때까지 카운터가 유지됩니다. 특히 임시 표현식에서 빌린 값을 let에 담을 때와 담지 않을 때의 생명주기가 다르므로 주의가 필요합니다.

EX 12 Ref<T> 생명주기 — 안전/위험/해결 패턴 + Ref::map
ref_lifetime.rs RUST 
use std::cell::{Ref, RefCell, RefMut};

#[derive(Debug)]
struct Stats {
    scores: Vec<i32>,
    name: String,
}

fn main() {
    let cell = RefCell::new(Stats {
        scores: vec![10, 20, 30],
        name: "player1".to_string(),
    });

    // ══ 패턴 1: 안전 — 스코프 안에서 borrow 완료 후 drop ══
    println!("=== 패턴 1: 스코프 자동 drop ===");
    {
        let r = cell.borrow();        // Ref 생성
        println!("scores: {:?}", r.scores);
    }                                  // Ref::drop() 호출 → 카운터 복원
    {
        let mut w = cell.borrow_mut(); // 이제 가능
        w.scores.push(40);
    }
    println!("scores: {:?}", cell.borrow().scores);

    // ══ 패턴 2: 임시 Ref가 즉시 drop되는 케이스 ══
    println!("\n=== 패턴 2: 임시 Ref ===");
    // 아래 줄에서 borrow()가 만든 Ref는 줄 끝에서 즉시 drop됨
    let len = cell.borrow().scores.len();  // Ref는 이미 사라짐
    println!("len = {}", len);
    // 바로 borrow_mut 가능 — Ref가 없으므로
    cell.borrow_mut().scores.push(50);

    // ══ 패턴 3: 위험 — Ref 변수 잡아두고 borrow_mut ══
    println!("\n=== 패턴 3: drop() 명시적 호출 ===");
    let r1 = cell.borrow();    // r1: Ref 살아있음
    println!("name: {}", r1.name);

    // 여기서 borrow_mut()하면 panic! — r1이 살아있으므로
    // cell.borrow_mut().scores.push(99);  ← PANIC

    drop(r1);   // 명시적 drop → 카운터 = 0
    cell.borrow_mut().scores.push(99);   // 이제 OK

    // ══ 패턴 4: Ref::map — 내부 필드만 빌리기 ══
    println!("\n=== 패턴 4: Ref::map ===");
    // 전체 Stats를 빌리는 대신 scores 필드만 Ref로 추출
    let scores_ref: Ref<Vec<i32>> =
        Ref::map(cell.borrow(), |s| &s.scores);
    println!("scores: {:?}", *scores_ref);
    drop(scores_ref);

    // ══ 패턴 5: RefMut::map — 내부 필드만 가변 빌리기 ══
    println!("\n=== 패턴 5: RefMut::map ===");
    let mut last: RefMut<i32> =
        RefMut::map(cell.borrow_mut(), |s| s.scores.last_mut().unwrap());
    *last *= 10;
    drop(last);

    println!("최종 scores: {:?}", cell.borrow().scores);
}

Rc<RefCell<T>> — 공유 소유권 + 내부 가변성

Rc<T>는 여러 소유자를 허용하지만 &T(불변 참조)만 제공합니다. 여기에 RefCell<T>를 결합하면 공유 소유권 + 내부 가변성을 동시에 얻을 수 있습니다. 공식 Book이 직접 소개하는 이 패턴은 단일 스레드에서 그래프, 트리, 공유 상태를 구현할 때 핵심적입니다.

By using Rc<RefCell<T>> we can have a value that can have multiple owners and that we can mutate! — The Rust Programming Language (공식 Book, Chapter 15.5)
타입소유권가변성 체크스레드
Box<T>단일컴파일 타임단일/멀티
Rc<T>다중 (불변)컴파일 타임단일 스레드만
Rc<RefCell<T>>다중 (가변)런타임단일 스레드만
Arc<Mutex<T>>다중 (가변)런타임멀티 스레드
EX 13 Rc<RefCell<T>> — 공유 노드 그래프 (공식 Book Cons List 확장)
rc_refcell_graph.rs RUST 
use std::cell::RefCell;
use std::rc::Rc;

// ── 공식 Book의 Cons List 변형 ──
#[derive(Debug)]
enum List {
    Cons(Rc<RefCell<i32>>, Rc<List>),
    Nil,
}

use List::{Cons, Nil};

// ── 간단한 공유 카운터 ──
#[derive(Debug, Clone)]
struct SharedCounter(Rc<RefCell<i32>>);

impl SharedCounter {
    fn new(v: i32) -> Self { SharedCounter(Rc::new(RefCell::new(v))) }
    fn inc(&self) { *self.0.borrow_mut() += 1; }
    fn get(&self) -> i32 { *self.0.borrow() }
    fn owners(&self) -> usize { Rc::strong_count(&self.0) }
}

fn main() {
    // ── 공식 Book 예제: Cons List ──
    let value = Rc::new(RefCell::new(5));

    let a = Rc::new(Cons(Rc::clone(&value), Rc::new(Nil)));
    let b = Cons(Rc::new(RefCell::new(3)), Rc::clone(&a));
    let c = Cons(Rc::new(RefCell::new(4)), Rc::clone(&a));

    // value를 수정하면 a, b, c 모두에 반영됨
    *value.borrow_mut() += 10;

    println!("a = {:?}", a);   // Cons(15, Nil)
    println!("b = {:?}", b);   // Cons(3, Cons(15, Nil))
    println!("c = {:?}", c);   // Cons(4, Cons(15, Nil))

    // ── 공유 카운터 ──
    println!("\n=== 공유 카운터 ===");
    let counter = SharedCounter::new(0);
    let c1 = counter.clone();  // Rc::clone — 데이터는 공유
    let c2 = counter.clone();

    counter.inc();
    c1.inc();
    c2.inc();
    c2.inc();

    // 모두 같은 값을 바라봄
    println!("counter={}, c1={}, c2={}",
        counter.get(), c1.get(), c2.get());   // 모두 4
    println!("소유자 수: {}", counter.owners());   // 3
}

OnceCell<T> / LazyCell<T> — 한 번만 초기화

공식 std::cell 문서는 Cell 타입을 네 가지로 분류합니다: Cell<T>, RefCell<T>, OnceCell<T>, LazyCell<T>.

OnceCell<T> is somewhat of a hybrid of Cell and RefCell that works for values that typically only need to be set once. This means that a reference &T can be obtained without moving or copying the inner value (unlike Cell) but also without runtime checks (unlike RefCell). However, its value can also not be updated once set unless you have a mutable reference to the OnceCell. — std::cell 공식 문서
타입초기화이후 수정런타임 비용멀티스레드
OnceCell<T> 처음 한 번 &mut OnceCell로만 없음 아님 (!Sync)
LazyCell<T, F> 첫 접근 시 자동 불가 없음 아님 (!Sync)
OnceLock<T> 처음 한 번 &mut OnceLock으로만 없음 가능 (Sync)
LazyLock<T, F> 첫 접근 시 자동 불가 없음 가능 (Sync)
EX 14 OnceCell / LazyCell — 지연 초기화 패턴
once_lazy_cell.rs RUST 
// OnceCell, LazyCell은 Rust 1.70+ stable
use std::cell::{LazyCell, OnceCell};

// ── OnceCell: 처음 한 번만 초기화, 이후 &T 반환 ──
struct Config {
    db_url: OnceCell<String>,
    max_conn: OnceCell<usize>,
}

impl Config {
    fn new() -> Self {
        Config {
            db_url: OnceCell::new(),
            max_conn: OnceCell::new(),
        }
    }

    // get_or_init: 이미 초기화됐으면 기존값 반환, 아니면 클로저 실행
    fn db_url(&self) -> &str {
        self.db_url.get_or_init(|| {
            println!("[초기화] DB URL 로딩...");
            "postgres://localhost/mydb".to_string()
        })
    }

    fn max_conn(&self) -> usize {
        *self.max_conn.get_or_init(|| {
            println!("[초기화] 커넥션 수 계산...");
            10
        })
    }

    // set: 이미 초기화됐으면 Err 반환
    fn override_url(&self, url: String) -> Result<(), String> {
        self.db_url.set(url).map_err(|e| format!("이미 초기화됨: {}", e))
    }
}

// ── LazyCell: 첫 접근 시 자동 초기화 ──
struct ExpensiveData {
    // 처음 접근할 때만 계산 — 그 전까지는 클로저만 저장
    computed: LazyCell<Vec<u32>>,
}

impl ExpensiveData {
    fn new() -> Self {
        ExpensiveData {
            computed: LazyCell::new(|| {
                println!("[LazyCell] 비싼 계산 실행 중...");
                (1..=5).map(|x| x * x).collect()
            }),
        }
    }

    fn get(&self) -> &Vec<u32> {
        &self.computed   // Deref — 첫 접근 시 자동 초기화
    }
}

fn main() {
    // ── OnceCell 예제 ──
    println!("=== OnceCell ===");
    let cfg = Config::new();

    // 첫 번째 호출: 초기화 클로저 실행
    println!("db_url = {}", cfg.db_url());
    // 두 번째 호출: 캐시된 값 반환 (클로저 실행 안 됨)
    println!("db_url = {}", cfg.db_url());

    // set 시도: 이미 초기화됐으므로 Err
    match cfg.override_url("mysql://...".to_string()) {
        Ok(_) => println!("변경 성공"),
        Err(e) => println!("변경 실패: {}", e),
    }
    println!("max_conn = {}", cfg.max_conn());

    // ── LazyCell 예제 ──
    println!("\n=== LazyCell ===");
    let data = ExpensiveData::new();
    println!("ExpensiveData 생성 완료 (아직 계산 안 됨)");

    // 처음 get() 호출 시 초기화
    println!("computed = {:?}", data.get());   // [비싼 계산] 실행
    println!("computed = {:?}", data.get());   // 캐시 반환
}

단일 스레드 한계 — 멀티스레드 환경의 대응 타입

Cell/RefCell/OnceCell/LazyCell은 모두 !Sync입니다. 즉, 여러 스레드에서 동시에 접근하면 컴파일 오류가 납니다. 공식 문서는 멀티스레드 환경에서는 Mutex<T>/RwLock<T>/OnceLock<T>를 사용하도록 안내합니다.

Cell<T>, RefCell<T>, and OnceCell<T> allow doing this in a single-threaded way—they do not implement Sync. If you need to do aliasing and mutation among multiple threads, Mutex<T>, RwLock<T>, OnceLock<T> or atomic types are the correct data structures to do so. — std::cell 공식 문서
단일 스레드멀티 스레드 대응차이
Cell<T> Atomic{Bool,I32,U64,...} 원자적 연산, lock-free
RefCell<T> Mutex<T> / RwLock<T> OS lock, 블로킹
Rc<T> Arc<T> 원자적 참조 카운팅
OnceCell<T> OnceLock<T> thread-safe 단일 초기화
LazyCell<T> LazyLock<T> thread-safe 지연 초기화
RefCell을 Arc로 감싸면?

Arc<RefCell<T>>는 컴파일 오류가 납니다. RefCell이 !Sync이므로 Arc의 T: Sync 요구사항을 충족하지 못합니다. 멀티스레드에서는 반드시 Arc<Mutex<T>>를 사용해야 합니다.
