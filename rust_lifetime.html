왜 라이프타임이 필요한가 — Dangling Reference 문제

Rust의 모든 참조(&T)에는 라이프타임이 있습니다. 라이프타임이란 그 참조가 유효한 코드 영역입니다. 라이프타임의 핵심 목적은 dangling reference(허상 참조)를 컴파일 타임에 차단하는 것입니다.

예제 1-1 — 기본 dangling reference (The Book Listing 10-16)

✕ 컴파일 에러
fn main() {
    let r;              // r 선언 (아직 아무것도 가리키지 않음)

    {
        let x = 5;      // x 생성
        r = &x;         // r이 x를 참조 — x는 이 블록이 끝나면 사라짐!
    }                   // ← x 여기서 drop

    println!("r: {r}"); // r이 가리키는 x는 이미 없음 → dangling reference!
}
error[E0597]: `x` does not live long enough
--> src/main.rs:6:13
5 | let x = 5;
| - binding `x` declared here
6 | r = &x;
| ^^ borrowed value does not live long enough
7 | }
| - `x` dropped here while still borrowed
9 | println!("r: {r}");
| - borrow later used here

예제 1-2 — 올바른 코드 (The Book Listing 10-18)

✓ 컴파일 성공
fn main() {
    let x = 5;            // x를 r보다 먼저 선언 → x가 더 오래 삶
    let r = &x;           // r이 x를 참조: x의 수명 ⊇ r의 수명 → OK
    println!("r: {r}");  // ✓ x가 아직 살아있음
}

예제 1-3 — C와의 비교 (왜 Rust가 필요한가)

참고: C언어 — 런타임 버그
// C 코드 (참고용, Rust 아님)
// int* r;
// { int x = 5; r = &x; }  ← x 스택에서 pop
// printf("%d", *r);         ← 💥 Undefined Behavior: 해제된 메모리 접근
//                              컴파일은 되지만 런타임에 프로그램이 터짐

// Rust는 동일한 상황을 E0597 에러로 컴파일 타임에 완전히 차단합니다

Borrow Checker 작동 원리

Rustonomicon에 따르면 모든 let 구문은 암묵적으로 스코프를 도입합니다. Borrow Checker는 이 스코프들을 분석해 참조가 항상 유효한지 확인합니다.

예제 2-1 — let 구문의 내부 (Rustonomicon)

// 우리가 쓰는 코드
let x = 0;
let y = &x;
let z = &y;

// 컴파일러가 내부적으로 보는 모습 (설명용 의사 코드)
// 'a: {
//     let x: i32 = 0;
//     'b: {
//         let y: &'b i32 = &'b x;  // 'b면 충분하므로 최소 'b 선택
//         'c: {
//             let z: &'c &'b i32 = &'c y;
//         }
//     }
// }

예제 2-2 — 외부 스코프로 참조 전달 시 라이프타임 확장 (Rustonomicon)

let x = 0;
let z;
let y = &x;
z = y;   // y를 z에 대입 → z의 라이프타임('b)에 맞춰 y도 'b로 추론됨

// 내부 표현:
// 'a: {
//   let x: i32 = 0;
//   'b: {
//     let z: &'b i32;
//     'c: {
//       let y: &'b i32 = &'b x;  // 'b scope로 전달되므로 'b로 추론
//       z = y;
//     }
//   }
// }

예제 2-3 — 가변 참조 vs 공유 참조 충돌 (Rustonomicon)

✕ 컴파일 에러
let mut data = vec![1, 2, 3];
let x = &data[0];     // 공유 참조 ('b 동안 유효해야 함)
data.push(4);         // ❌ &mut data 필요 — 그러나 x(&data)가 아직 'b 중!
println!("{}", x);   // x가 여기까지 살아있어야 하므로 'b 안 끝남
✓ 해결 — x의 마지막 사용 이후에 push
let mut data = vec![1, 2, 3];
let x = &data[0];
println!("{}", x);   // ✓ x 마지막 사용 → 이후 borrow 종료 (NLL)
data.push(4);        // ✓ 가변 참조 가능

예제 2-4 — Non-Lexical Lifetimes (NLL, Rust 2018+)

라이프타임은 변수 스코프 끝이 아닌 마지막 실제 사용까지만 유지됩니다.

✓ 컴파일 성공 (NLL 덕분)
let mut data = vec![1, 2, 3];
let mut x = &data[0];

println!("{}", x);  // 첫 번째 borrow 마지막 사용 → 이후 종료
data.push(4);       // ✓ NLL: 이전 borrow 이미 종료됨
x = &data[3];        // ✓ 새로운 borrow 시작
println!("{}", x);

'a 애노테이션 기본 문법

중요 (The Book §10.3): 라이프타임 애노테이션은 참조가 실제로 살아있는 시간을 바꾸지 않습니다. 여러 참조의 라이프타임 관계를 컴파일러에게 설명하는 것뿐입니다.

예제 3-1 — 기본 문법 형태 (The Book §10.3)

// & 바로 뒤, 타입 앞에 라이프타임 표기
&i32              // 라이프타임 없는 참조 (컴파일러가 추론)
&'a i32           // 라이프타임 'a를 가진 참조
&'a mut i32       // 라이프타임 'a를 가진 가변 참조
&'static str     // 'static 라이프타임 (프로그램 전체)

// 함수: 꺾쇠괄호 안에 선언 (제네릭 T와 같은 위치)
fn foo<'a>(x: &'a i32) { }

// 여러 라이프타임
fn bar<'a, 'b>(x: &'a i32, y: &'b i32) { }

// 라이프타임 + 제네릭 타입 함께
fn baz<'a, T>(x: &'a T) { }

예제 3-2 — print_refs + failed_borrow (Rust By Example)

✓ 컴파일 성공
// 서로 다른 라이프타임 'a, 'b — 함수 실행 중 둘 다 유효해야 함
fn print_refs<'a, 'b>(x: &'a i32, y: &'b i32) {
    println!("x is {} and y is {}", x, y);
}

fn main() {
    let (four, nine) = (4, 9);
    print_refs(&four, &nine); // ✓ 두 라이프타임 모두 유효
}
✕ 컴파일 에러
// 'a는 호출자가 결정 — 함수 내부 지역변수보다 길 수 있음
// 짧은 라이프타임을 더 긴 것으로 강제 불가!
fn failed_borrow<'a>() {
    let _x = 12;
    let _y: &'a i32 = &_x; // ❌ _x의 라이프타임 < 'a
}

함수에서 라이프타임 — 입력·출력 참조

함수가 참조를 입력받아 참조를 반환할 때, 컴파일러는 반환된 참조가 어느 입력에서 왔는지 알아야 합니다. 이를 모르면 반환값의 유효 기간을 판단할 수 없습니다.

예제 4-1 — 라이프타임 없이 참조 반환 → 에러 (Listing 10-20)

✕ 컴파일 에러
fn longest(x: &str, y: &str) -> &str { // 반환값이 x인지 y인지 불명!
    if x.len() > y.len() { x } else { y }
}
error[E0106]: missing lifetime specifier
  = help: this function's return type contains a borrowed value,
            but the signature does not say whether it is borrowed from `x` or `y`
help: consider: fn longest<'a>(x: &'a str, y: &'a str) -> &'a str

예제 4-2 — 라이프타임 추가로 해결 (Listing 10-21)

✓ 컴파일 성공
// 'a = x와 y 중 더 짧은 쪽의 라이프타임. 반환값은 그보다 오래 살 수 없음.
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}

fn main() {
    let s1 = String::from("abcd");
    let s2 = "xyz";  // &'static str

    let result = longest(s1.as_str(), s2);
    println!("The longest string is {result}"); // ✓ "abcd"
}

예제 4-3 — result 사용 위치가 결과를 바꿈 (Listing 10-22 vs 10-23)

✓ 성공 — result를 inner scope 안에서 사용
fn main() {
    let string1 = String::from("long string is long");
    {
        let string2 = String::from("xyz"); // 'a = 이 inner scope
        let result = longest(string1.as_str(), string2.as_str());
        println!("{result}"); // ✓ inner 안에서 사용 → OK
    }
}
✕ 에러 — result를 inner scope 밖에서 사용
fn main() {
    let string1 = String::from("long string is long");
    let result;
    {
        let string2 = String::from("xyz");
        result = longest(string1.as_str(), string2.as_str());
    } // string2 drop! 'a(=inner) 끝남
    println!("{result}"); // ❌ 'a 이후 사용
}

예제 4-4 — 반환값과 무관한 파라미터는 'a 불필요 (The Book §10.3)

✓ 컴파일 성공
// x만 반환 → y의 라이프타임은 반환값과 무관
fn first_param<'a>(x: &'a str, y: &str) -> &'a str { x }

fn main() {
    let s1 = String::from("hello");
    let result;
    {
        let s2 = String::from("world");
        result = first_param(s1.as_str(), s2.as_str()); // ✓ y는 무관
    }
    println!("{result}"); // ✓ result는 s1에서 왔으므로 유효
}

예제 4-5 — 지역변수 참조 반환 → 항상 에러 (The Book §10.3)

✕ 컴파일 에러
fn dangle<'a>(x: &str, y: &str) -> &'a str {
    let result = String::from("really long string");
    result.as_str() // ❌ result는 함수 끝에서 drop → dangling reference!
}
// error[E0515]: cannot return value referencing local variable `result`
✓ 해결 — 소유 타입 반환
fn no_dangle() -> String {
    String::from("really long string")  // ✓ 소유권 이전 → 호출자가 관리
}
fn main() {
    let s = no_dangle(); println!("{s}"); // ✓
}

예제 4-6 — Rust By Example의 함수 라이프타임 패턴들

✓ 컴파일 성공
// 1) 입력 하나
fn print_one<'a>(x: &'a i32) { println!("{}", x); }

// 2) 가변 참조도 라이프타임 적용 가능
fn add_one<'a>(x: &'a mut i32) { *x += 1; }

// 3) 서로 다른 라이프타임 두 개
fn print_multi<'a, 'b>(x: &'a i32, y: &'b i32) {
    println!("x={}, y={}", x, y);
}

// 4) 입력 중 하나만 반환에 연결
fn pass_x<'a, 'b>(x: &'a i32, _: &'b i32) -> &'a i32 { x }

fn main() {
    let (x, y) = (7, 9);
    print_one(&x);
    print_multi(&x, &y);
    let z = pass_x(&x, &y);
    print_one(z);
    let mut t = 3;
    add_one(&mut t);
    print_one(&t);
}

예제 4-7 — first_word: 생략 vs 명시 (Listing 10-25)

✓ 두 버전 모두 동일한 의미
// 생략 버전 — 생략 규칙 덕분에 라이프타임 불필요
fn first_word(s: &str) -> &str {
    let bytes = s.as_bytes();
    for (i, &item) in bytes.iter().enumerate() {
        if item == b' ' { return &s[0..i]; }
    }
    &s[..]
}

// 명시 버전 — 완전히 같은 의미
fn first_word_explicit<'a>(s: &'a str) -> &'a str {
    let bytes = s.as_bytes();
    for (i, &item) in bytes.iter().enumerate() {
        if item == b' ' { return &s[0..i]; }
    }
    &s[..]
}

fn main() {
    let s = String::from("hello world");
    println!("{}", first_word(&s[..]));          // "hello"
    println!("{}", first_word("hello world"));   // "hello" (리터럴도 OK)
}

구조체에서 라이프타임

예제 5-1 — 튜플·일반·enum 구조체 (Rust By Example)

✓ 컴파일 성공
#[derive(Debug)]
struct Borrowed<'a>(&'a i32);              // 튜플 구조체

#[derive(Debug)]
struct NamedBorrowed<'a> {               // 일반 구조체
    x: &'a i32,
    y: &'a i32,
}

#[derive(Debug)]
enum Either<'a> {
    Num(i32),          // 소유값: 라이프타임 불필요
    Ref(&'a i32),     // 참조: 라이프타임 필요
}

fn main() {
    let (x, y) = (18, 15);
    let single    = Borrowed(&x);
    let double    = NamedBorrowed { x: &x, y: &y };
    let reference = Either::Ref(&x);
    let number    = Either::Num(y);  // y 복사, 참조 아님

    println!("{:?}", single);    // Borrowed(18)
    println!("{:?}", double);    // NamedBorrowed { x: 18, y: 15 }
    println!("{:?}", reference); // Ref(18)
    println!("{:?}", number);    // Num(15)
}

예제 5-2 — ImportantExcerpt (Listing 10-24)

✓ 컴파일 성공
struct ImportantExcerpt<'a> {
    part: &'a str,
    // 의미: 이 인스턴스는 part가 가리키는 값보다 오래 살 수 없다
}

fn main() {
    let novel = String::from("Call me Ishmael. Some years ago...");
    // novel이 먼저 선언 → i 보다 오래 삶 → OK
    let first = novel.split('.').next().unwrap();
    let i = ImportantExcerpt { part: first };
    println!("발췌: {}", i.part); // ✓
}

예제 5-3 — impl 블록에서의 라이프타임 + 메서드 (The Book §10.3)

✓ 컴파일 성공
impl<'a> ImportantExcerpt<'a> {
    // 반환이 i32(참조 아님) → 라이프타임 불필요
    fn level(&self) -> i32 { 3 }

    // 생략 규칙 3 적용: &self 있음 → 출력이 self의 라이프타임에 연결
    // 컴파일러 내부: fn announce<'a,'b>(&'a self, ann: &'b str) -> &'a str
    fn announce_and_return_part(&self, announcement: &str) -> &str {
        println!("Attention: {announcement}");
        self.part
    }
}

fn main() {
    let novel = String::from("Call me Ishmael. Some years ago...");
    let first = novel.split('.').next().unwrap();
    let i = ImportantExcerpt { part: first };
    println!("Level: {}", i.level());
    println!("Part: {}", i.announce_and_return_part("들어주세요!"));
}

예제 5-4 — 소유 vs 참조 구조체 비교

// 소유 타입만 → 라이프타임 불필요
struct ConfigOwned { name: String, port: u16 }

// 참조 포함 → 라이프타임 필수
struct ConfigRef<'a> { name: &'a str, port: u16 }

fn main() {
    // ConfigOwned: 자유롭게 이동, 라이프타임 걱정 없음
    let c1 = ConfigOwned { name: String::from("localhost"), port: 8080 };
    println!("{}:{}", c1.name, c1.port);

    // ConfigRef: host보다 짧게 살아야 함 (자동 보장)
    let host = String::from("localhost");
    let c2 = ConfigRef { name: &host, port: 8080 };
    println!("{}:{}", c2.name, c2.port);
}

생략 규칙 (Elision Rules)

(The Book §10.3): 생략 규칙은 프로그래머가 따를 규칙이 아닙니다. 컴파일러가 검사하는 특정 패턴들의 집합입니다. 규칙을 적용해도 불명확하면 추측 없이 에러를 냅니다.

규칙 1 — 각 입력 참조는 개별 라이프타임

fn foo(x: &i32)               // → fn foo<'a>(x: &'a i32)
fn bar(x: &i32, y: &i32)      // → fn bar<'a,'b>(x: &'a i32, y: &'b i32)

규칙 2 — 입력 라이프타임이 정확히 1개면 출력에도 적용

fn foo(x: &i32) -> &i32                     // → fn foo<'a>(x: &'a i32) -> &'a i32
fn substr(s: &str, n: usize) -> &str       // → fn substr<'a>(s: &'a str, n: usize) -> &'a str
// (usize는 참조가 아니므로 입력 라이프타임 카운트에 포함되지 않음)

규칙 3 — 메서드에서 &self가 있으면 출력은 self의 라이프타임

fn get_mut(&mut self) -> &mut i32
// → fn get_mut<'a>(&'a mut self) -> &'a mut i32

fn announce(&self, ann: &str) -> &str
// → fn announce<'a,'b>(&'a self, ann: &'b str) -> &'a str

예제 6-1 — 생략 가능한 패턴들 (Rust Reference)

✓ 모두 컴파일 성공
fn print1(s: &str);                              // 생략
fn print2(s: &'_ str);                          // '_ 플레이스홀더 (생략과 동일)
fn print3<'a>(s: &'a str);                      // 명시 (모두 동일한 의미)

fn debug1(lvl: usize, s: &str);                 // usize 제외, 참조 1개
fn debug2<'a>(lvl: usize, s: &'a str);         // 동일

fn substr1(s: &str, until: usize) -> &str;     // 생략
fn substr2<'a>(s: &'a str, until: usize) -> &'a str; // 동일

예제 6-2 — 생략 불가 케이스 (Rust Reference — "ILLEGAL")

✕ 컴파일 에러
fn get_str() -> &str;                         // ❌ 파라미터 없어 추론 불가
fn frob(s: &str, t: &str) -> &str;           // ❌ 입력 2개, 어느 쪽인지 불명

// 해결: 명시적으로 지정
fn frob<'a>(s: &'a str, t: &'a str) -> &'a str; // ✓

// ─── first_word 추적 ───────────────────────────────

7

'static 라이프타임

(The Book §10.3 경고): 에러 메시지가 'static을 쓰라고 제안해도, 대부분은 dangling reference 또는 라이프타임 불일치가 진짜 원인입니다. 'static을 추가하기 전에 그 문제를 먼저 고치세요.

예제 7-1 — 문자열 리터럴과 static 변수 (Rust By Example)

✓ 컴파일 성공
static NUM: i32 = 18;  // 전역 static — 바이너리에 저장, 프로그램 내내 유효

// NUM('static)을 입력의 라이프타임('a)으로 강제(coerce) 가능
fn coerce_static<'a>(_: &'a i32) -> &'a i32 { &NUM }

fn main() {
    {
        let static_string = "I'm in read-only memory";
        println!("static_string: {static_string}");
        // 이 스코프 종료 후에도 데이터는 바이너리에 남아있음
    }
    {
        let lifetime_num = 9;
        let coerced = coerce_static(&lifetime_num);
        println!("coerced_static: {coerced}");
    }
    println!("NUM: {} stays accessible!", NUM);
}

예제 7-2 — 참조 lifetime vs 트레잇 바운드 (Rust By Example)

// ① &'static T — 참조의 라이프타임
let s: &'static str = "hello world";  // 리터럴은 자동으로 'static

// ② T: 'static — 트레잇 바운드 (T가 'static보다 짧은 참조를 포함 안 함)
use std::fmt::Debug;
fn print_it(input: impl Debug + 'static) {
    println!("'static value: {:?}", input);
}

fn main() {
    let i = 5;
    print_it(i);         // ✓ i32는 소유 타입, 참조 없음 → 'static 충족
    print_it("hi");      // ✓ &'static str → 충족
    print_it(String::from("owned")); // ✓ String 소유 타입 → 충족
    print_it(&i);        // ❌ &i의 수명 = main 스코프 → 'static 아님
}

예제 7-3 — 함수에서 'static 반환 패턴

✓ 컴파일 성공
// 리터럴은 'static이므로 반환 가능
fn get_greeting() -> &'static str { "Hello!" }

fn get_error(code: u32) -> &'static str {
    match code {
        404 => "Not Found",
        500 => "Internal Server Error",
        _   => "Unknown Error",
    }
}

fn main() {
    println!("{}", get_greeting());
    println!("{}", get_error(404));
}

예제 7-4 — Box::leak으로 동적 'static 생성 (Rust By Example)

⚠ 고급 패턴 — 의도적 메모리 누수
fn make_static(s: String) -> &'static str {
    Box::leak(s.into_boxed_str())
    // 이 메모리는 프로그램 종료까지 해제 안 됨!
    // 꼭 필요한 경우에만 사용 (예: 전역 설정, 스레드 공유)
}

fn main() {
    let dynamic = String::from("dynamically created");
    let r: &'static str = make_static(dynamic);
    println!("{r}");  // 프로그램 어디서든 유효
}

제네릭 + 트레잇 바운드 + 라이프타임 함께 쓰기

예제 8-1 — 세 가지 조합 (The Book §10.3 마지막 예제)

✓ 컴파일 성공
use std::fmt::Display;

fn longest_with_announcement<'a, T>(  // 'a + T 함께 선언
    x:   &'a str,
    y:   &'a str,
    ann: T,             // 소유 타입 → 라이프타임 불필요
) -> &'a str
where
    T: Display,        // where절로 트레잇 바운드 분리
{
    println!("Announcement: {ann}");
    if x.len() > y.len() { x } else { y }
}

fn main() {
    let s1 = String::from("abcd");
    let s2 = "xyz";
    let result = longest_with_announcement(
        s1.as_str(), s2,
        "Today is someone's birthday!",
    );
    println!("The longest string is {result}"); // ✓ "abcd"
}

예제 8-2 — 트레잇 구현에서 라이프타임 (Rust By Example)

✓ 컴파일 성공
#[derive(Debug)]
struct Borrowed<'a> { x: &'a i32 }

impl<'a> std::fmt::Display for Borrowed<'a> {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        write!(f, "Borrowed({})", self.x)
    }
}

impl<'a> Default for Borrowed<'a> {
    fn default() -> Self {
        static ZERO: i32 = 0;
        Borrowed { x: &ZERO }
    }
}

fn main() {
    let x = 42;
    let b = Borrowed { x: &x };
    println!("{b}");                     // Borrowed(42)
    println!("{b:?}");                   // Borrowed { x: 42 }
    println!("{}", Borrowed::default()); // Borrowed(0)
}

예제 8-3 — 라이프타임 포함 관계 'a: 'b (Rust Reference)

✓ 컴파일 성공
// 'a: 'b 의미: 'a가 최소 'b만큼 살아있음 → &'a를 &'b로 coerce 가능
fn longer<'a: 'b, 'b>(x: &'a i32, mut y: &'b i32) {
    y = x;  // 'a: 'b이므로 x(&'a)를 y(&'b)에 대입 가능
    println!("{y}");
}

fn main() {
    longer(&10i32, &5i32); // ✓
}

자주 만나는 패턴 & 실수 모음

예제 9-1 — 루프에서의 라이프타임 재사용 (Rustonomicon)

✓ 컴파일 성공
let mut data = vec![1, 2, 3];
let mut x = &data[0];

println!("{}", x);   // 첫 번째 borrow 마지막 사용
data.push(4);        // ✓ NLL: 이전 borrow 종료됨
x = &data[3];         // ✓ 새 borrow 시작
println!("{}", x);

예제 9-2 — Drop이 있는 구조체 (Rustonomicon)

✕ 컴파일 에러
#[derive(Debug)]
struct WithDrop<'a>(&'a i32);
impl Drop for WithDrop<'_> { fn drop(&mut self) {} }

let mut data = vec![1, 2, 3];
let x = WithDrop(&data[0]);
println!("{:?}", x);
data.push(4);   // ❌ x의 drop은 스코프 끝 → 아직 borrow 중!
✓ 해결 — drop을 명시적으로 먼저 호출
drop(x);         // ✓ x를 여기서 명시 drop → borrow 종료
data.push(4);   // ✓ 가변 참조 가능

예제 9-3 — if 분기에서의 라이프타임 (Rustonomicon)

✓ 컴파일 성공
let mut data = vec![1, 2, 3];
let x = &data[0];

if true {
    println!("{}", x);  // ✓ x의 마지막 사용 (이 분기에서)
    data.push(4);       // ✓ 이미 마지막 사용 후이므로 OK
} else {
    // 이 분기에서 x 미사용 → 생성 시점이 마지막 사용
    data.push(5);       // ✓
}

예제 9-4 — 흔한 실수 3가지 + 해결책

✕ 실수 1 — 지역변수 참조 반환
fn bad1<'a>() -> &'a str {
    let s = String::from("hello");
    &s  // ❌ s는 함수 끝에서 drop됨
}
// 해결: fn good1() -> String { String::from("hello") }
✕ 실수 2 — 구조체 라이프타임 누락
struct Config { name: &str }  // ❌ error[E0106]: missing lifetime specifier
// 해결 1: struct Config<'a> { name: &'a str }
// 해결 2: struct Config { name: String }  ← 대부분의 경우 이게 나음
✕ 실수 3 — 입력 없이 참조 반환
fn get_str() -> &str;  // ❌ 추론 불가 + dangling reference가 될 것
// 해결: fn get_str() -> &'static str { "literal" }
// 또는: fn get_str() -> String { "hello".to_string() }

예제 9-5 — 실용 패턴: 제로 카피 파서

✓ 컴파일 성공
// HTTP 헤더를 String 복사 없이 파싱 — 라이프타임으로 안전성 보장
struct ParsedHeader<'a> {
    key:   &'a str,
    value: &'a str,
}

fn parse_header<'a>(line: &'a str) -> Option<ParsedHeader<'a>> {
    let mut parts = line.splitn(2, ": ");
    Some(ParsedHeader { key: parts.next()?, value: parts.next()? })
    // key, value는 line의 슬라이스 → String 복사 없이 O(1)
    // 'a 덕분에 컴파일러가 "line이 살아있는 동안만 사용"을 보장
}

fn main() {
    let line = "Content-Type: application/json";
    if let Some(h) = parse_header(line) {
        println!("key={}, value={}", h.key, h.value);
        // key="Content-Type"  value="application/json"
    }
}

상황별 라이프타임 명시 요약표

상황라이프타임이유
fn f(x: &str)불필요반환값 없음
fn f(x: &str) -> &str불필요 (규칙 2)입력 참조 1개 → 출력 자동 연결
fn f(x: &str, n: usize) -> &str불필요 (규칙 2)usize 제외, 참조 입력 1개
fn f(x: &str, y: &str) -> &str필요입력 2개 → 어느 쪽인지 불명
fn f(&self) -> &str불필요 (규칙 3)&self 있음 → 출력이 self에 연결
fn f(&self, x: &str) -> &str불필요 (규칙 3)&self 있음 → 출력이 self에 연결
struct Foo { x: &str }필요참조 필드 → 항상 명시
struct Foo { x: String }불필요소유 타입
const S: &str = "hi"불필요 (암묵적 'static)const/static은 자동 'static
fn f() -> &str항상 에러입력 없이 참조 반환 → 추론 불가

핵심 원칙 5가지

  • 라이프타임은 수명을 바꾸지 않는다. 참조 간의 관계를 컴파일러에게 알려주는 애노테이션일 뿐입니다.
  • 'a = 더 짧은 쪽. fn f<'a>(x: &'a str, y: &'a str)에서 'a는 x와 y 중 더 짧게 사는 쪽으로 실체화됩니다.
  • 함수 내부 지역변수 참조는 절대 반환 불가. 반환 참조는 반드시 입력 파라미터 중 하나에 연결되어야 합니다.
  • 생략 규칙이 해결 못하면 명시해야 한다. 컴파일러는 추측하지 않습니다.
  • 'static 남용 주의. 대부분의 'static 에러 제안은 라이프타임 불일치가 진짜 원인입니다.
