§ 01

parse() — 문자열에서 값을 꺼내는 법

parse()는 &str을 다른 타입으로 변환하는 메서드입니다. 핵심은 항상 Result<T, E>를 반환한다는 것 — 파싱이 실패할 수 있기 때문입니다.

Rust
 1fn main() {
 2    // &str → i32 파싱
 3    let n: i32 = "42".parse().unwrap();
 4    println!("{}", n); // 42
 5
 6    // 터보피시로 타입 명시
 7    let m = "100".parse::<i32>().unwrap();
 8    println!("{}", m); // 100
 9
10    // Result로 처리
11    let r: Result<i32, _> = "abc".parse();
12    println!("{:?}", r);
13    // Err(ParseIntError { kind: InvalidDigit })
14}
📌 왜 Result를 반환하나
문자열은 어떤 내용이든 담을 수 있습니다. "abc"를 i32로 바꾸는 것은 불가능하므로 실패 가능성을 타입으로 표현합니다. 이것이 Rust가 런타임 예외 대신 Result를 사용하는 이유입니다.
📌 FromStr 트레이트
parse()는 FromStr 트레이트를 구현한 타입이면 모두 사용 가능합니다. 표준 라이브러리의 i32, f64, bool, char, String 등이 모두 구현되어 있고, 직접 구현할 수도 있습니다.
§ 02

터보피시 :: — 타입을 명시하는 법

parse()는 어떤 타입으로 변환할지 알아야 합니다. ::<T> 문법(터보피시)으로 직접 명시하거나, 변수 타입 선언으로 컴파일러가 추론하게 할 수 있습니다.

Rust
 1fn main() {
 2    // 터보피시 필수 — 타입 추론 불가
 3    let v = "42".parse::<i32>().unwrap();
 4    println!("{}", v);
 5
 6    // 타입 추론 가능 — 터보피시 생략
 7    let n: i32 = "42".parse().unwrap();
 8    println!("{}", n);
 9
10    // 벡터에 넣을 때 — 추론 가능
11    let nums: Vec<i32> = vec!["1","2","3"]
12        .iter()
13        .map(|s| s.parse().unwrap())
14        .collect();
15    println!("{:?}", nums); // [1, 2, 3]
16}
📌 터보피시라는 이름의 유래
::<> 모양이 물고기처럼 생겼다고 해서 "turbofish"라는 별명이 붙었습니다. Rust 커뮤니티에서 공식적으로 사용되는 용어입니다.
⚠ 추론이 불가한 경우
let v = "42".parse().unwrap();처럼 타입 컨텍스트가 없으면 컴파일 에러가 납니다. 이럴 때 터보피시 .parse::<i32>() 또는 변수에 타입 명시 let v: i32 = ... 중 하나를 선택합니다.
§ 03

Result — 성공과 실패를 담는 타입

Result<T, E>는 두 가지 상태만 가집니다: Ok(T) 성공값, Err(E) 에러값. C언어의 에러 코드나 Java의 예외와 달리, 타입 시스템이 처리를 강제합니다.

Rust
 1fn main() {
 2    // Ok — 성공
 3    let ok: Result<i32, _> = "42".parse();
 4    println!("{:?}", ok); // Ok(42)
 5
 6    // Err — 실패
 7    let err: Result<i32, _> = "abc".parse();
 8    println!("{:?}", err);
 9    // Err(ParseIntError { kind: InvalidDigit })
10
11    // is_ok / is_err
12    println!("{}", ok.is_ok());  // true
13    println!("{}", err.is_err()); // true
14}
📌 Result는 값이다
Java의 try/catch와 달리 Result는 일반 값입니다. 함수 인자로 전달하고, 벡터에 저장하고, 변환하고, 반환할 수 있습니다. 에러 처리가 제어 흐름이 아닌 데이터 흐름의 일부가 됩니다.
📌 Err를 무시하면 경고
let _ = "abc".parse::<i32>();처럼 Result를 사용하지 않으면 컴파일러가 unused Result that must be used 경고를 냅니다. 실수로 에러를 무시하는 것을 방지합니다.
§ 04

숫자 파싱 — i32, u8, f64 타입별 차이

Rust의 숫자 타입은 각각 허용 범위가 다릅니다. 같은 문자열이라도 목표 타입에 따라 파싱 성공 여부가 달라집니다.

Rust
 1fn main() {
 2    // i32: -2147483648 ~ 2147483647
 3    let a = "2147483647".parse::<i32>();
 4    println!("{:?}", a); // Ok(2147483647)
 5
 6    // u8: 0 ~ 255
 7    let b = "255".parse::<u8>();
 8    println!("{:?}", b); // Ok(255)
 9
10    // f64: 소수점 OK
11    let c = "3.14".parse::<f64>();
12    println!("{:?}", c); // Ok(3.14)
13
14    // bool
15    let d = "true".parse::<bool>();
16    println!("{:?}", d); // Ok(true)
17}
📌 타입별 허용 범위
i8 -128~127 · i32 -2147483648~2147483647 · u8 0~255 · u32 0~4294967295 · f32/f64 소수점 OK · bool "true"/"false"만 허용
⚠ 정수와 소수점
"3.14".parse::<i32>()는 실패합니다 — 정수 파서는 소수점을 이해하지 못합니다. 소수점이 있으면 f64로, 정수면 i32/i64로 파싱해야 합니다.
Rust
 1fn main() {
 2    // 범위 초과 (u8 max=255)
 3    let over = "256".parse::<u8>();
 4    println!("{:?}", over);
 5    // Err(ParseIntError { kind: PosOverflow })
 6
 7    // 음수를 부호 없는 타입으로
 8    let neg = "-1".parse::<u8>();
 9    println!("{:?}", neg);
10    // Err(ParseIntError { kind: InvalidDigit })
11
12    // f64에 정수 파서 사용
13    let float = "3.14".parse::<i32>();
14    println!("{:?}", float);
15    // Err(ParseIntError { kind: InvalidDigit })
16}
§ 05

파싱 실패 케이스 완전 정리

파싱이 실패하는 경우는 크게 4가지입니다: 빈 문자열, 공백 포함, 허용 범위 초과, 잘못된 문자. 각각 다른 IntErrorKind를 반환합니다.

Rust
 1fn main() {
 2    // 빈 문자열
 3    let e1 = "".parse::<i32>();
 4    println!("{:?}", e1);
 5    // Err(ParseIntError { kind: Empty })
 6
 7    // 공백 포함
 8    let e2 = " 42".parse::<i32>();
 9    println!("{:?}", e2);
10    // Err(ParseIntError { kind: InvalidDigit })
11
12    // trim 후 파싱 — 공백 제거
13    let e3 = " 42 ".trim().parse::<i32>();
14    println!("{:?}", e3); // Ok(42)
15
16    // 문자 섞임
17    let e4 = "42px".parse::<i32>();
18    println!("{:?}", e4);
19    // Err(ParseIntError { kind: InvalidDigit })
20}
⚠ 공백은 자동으로 처리되지 않는다
" 42", "42 "처럼 앞뒤 공백이 있으면 파싱이 실패합니다. 사용자 입력을 파싱할 때는 반드시 .trim()을 먼저 호출하세요.
📌 에러 종류 확인
ParseIntError::kind()로 에러 원인을 구분할 수 있습니다: Empty(빈 문자열), InvalidDigit(잘못된 문자), PosOverflow/NegOverflow(범위 초과).
✗ 흔한 실수
"42".parse::<i32>()는 성공하지만 "42 ".parse::<i32>()는 실패합니다. 파일에서 읽은 줄에는 개행문자가 붙어 있으므로 반드시 .trim() 처리를 해야 합니다.
§ 06

match로 Ok/Err 처리 — 가장 명시적인 방법

match는 Ok와 Err 두 케이스를 모두 명시적으로 처리합니다. 에러 종류까지 세분화해 처리해야 할 때 가장 적합합니다.

Rust
 1fn main() {
 2    let input = "42";
 3
 4    // match — 가장 명시적
 5    match input.parse::<i32>() {
 6        Ok(n)  => println!("파싱 성공: {}", n),
 7        Err(e) => println!("파싱 실패: {}", e),
 8    }
 9
10    // 에러 종류 구분
11    match "abc".parse::<i32>() {
12        Ok(n) => println!("{}", n),
13        Err(e) => {
14            println!("에러 종류: {:?}", e.kind());
15            // InvalidDigit
16        }
17    }
18}
📌 match는 완전해야 한다
Rust의 match는 모든 경우를 반드시 처리해야 컴파일됩니다. Ok만 쓰고 Err를 생략하면 컴파일 에러가 납니다. 이것이 에러를 놓치는 실수를 원천 차단합니다.
Rust
 1use std::num::IntErrorKind;
 2fn main() {
 3    let inputs = ["42", "", "999", "abc"];
 4    for s in &inputs {
 5        match s.parse::<u8>() {
 6            Ok(n) => println!("{} → {}", s, n),
 7            Err(e) => match e.kind() {
 8                IntErrorKind::Empty =>
 9                    println!("{} → 빈 문자열", s),
10                IntErrorKind::PosOverflow =>
11                    println!("{} → 범위 초과", s),
12                _ =>
13                    println!("{} → 잘못된 형식", s),
14            }
15        }
16    }
17}
📌 IntErrorKind로 에러 분류
e.kind()로 에러 원인을 분기할 수 있습니다. 사용자에게 구체적인 에러 메시지를 보여줄 때 유용합니다: 빈 입력인지, 숫자 범위 초과인지, 형식이 잘못됐는지.
§ 07

unwrap · expect · unwrap_or — 간편 처리 메서드

매번 match를 쓰기 번거로울 때 사용하는 메서드들입니다. 각각 트레이드오프가 다릅니다.

Rust
 1fn main() {
 2    // unwrap — Err면 패닉
 3    let n = "42".parse::<i32>().unwrap();
 4    println!("{}", n); // 42
 5
 6    // expect — 패닉 메시지 지정
 7    let m = "42".parse::<i32>()
 8        .expect("숫자여야 합니다");
 9    println!("{}", m); // 42
10
11    // unwrap_or — 기본값
12    let a = "abc".parse::<i32>().unwrap_or(0);
13    println!("{}", a); // 0
14
15    // unwrap_or_else — 클로저로 기본값
16    let b = "abc".parse::<i32>()
17        .unwrap_or_else(|_| -1);
18    println!("{}", b); // -1
19}
⚠ unwrap은 프로덕션에서 자제
unwrap()은 Err일 때 즉시 패닉(프로그램 종료)을 일으킵니다. 프로토타입이나 테스트 코드에서는 편하지만, 실제 서비스 코드에서는 ? 연산자나 match를 사용하세요.
📌 expect로 디버깅 향상
unwrap() 대신 expect("설명")을 사용하면 패닉 메시지에 컨텍스트가 추가됩니다. 어느 파싱에서 실패했는지 즉시 알 수 있어 디버깅이 쉬워집니다.
✓ unwrap_or_else 활용
기본값이 비용이 큰 경우(힙 할당, I/O 등) unwrap_or 대신 unwrap_or_else(|| ...)를 사용하세요. 클로저는 Err일 때만 실행됩니다.
§ 08

if let · while let — 간결한 패턴

Ok인 경우만 처리하거나, 루프 안에서 파싱할 때 if let과 while let을 사용하면 코드가 훨씬 간결해집니다.

Rust
 1fn main() {
 2    let input = "42";
 3
 4    // if let — Ok만 처리
 5    if let Ok(n) = input.parse::<i32>() {
 6        println!("파싱됨: {}", n);
 7    }
 8
 9    // if let else — Ok/Err 모두
10    if let Ok(n) = input.parse::<i32>() {
11        println!("성공: {}", n);
12    } else {
13        println!("실패");
14    }
15
16    // while let — 루프에서 파싱
17    let mut items = vec!["1","x","2","3"];
18    items.reverse();
19    while let Some(s) = items.pop() {
20        if let Ok(n) = s.parse::<i32>() {
21            println!("숫자: {}", n);
22        }
23    }
24}
📌 if let vs match
if let Ok(n) = ...은 Ok인 경우만 처리하고 Err는 조용히 무시합니다. 에러를 정말 무시해도 괜찮을 때만 사용하고, 에러 처리가 필요하면 else 분기를 추가하거나 match를 사용합니다.
⚠ Err 무시 주의
if let Ok(n) = input.parse::<i32>() { ... }에서 파싱이 실패하면 블록이 조용히 실행되지 않습니다. 의도한 동작인지 항상 확인하세요.
§ 09

? 연산자 — 에러를 호출자에게 전파

함수 안에서 파싱 결과를 처리하다가 실패하면 즉시 함수를 종료하고 싶을 때 ?를 사용합니다. match의 축약형입니다.

Rust
 1fn parse_port(s: &str) -> Result<u16, String> {
 2    // ? — Err면 즉시 반환
 3    let n: u16 = s.parse()
 4        .map_err(|e| format!("포트 오류: {}", e))?;
 5    if n == 0 {
 6        return Err("포트는 0이 될 수 없음".into());
 7    }
 8    Ok(n)
 9}
10fn main() {
11    println!("{:?}", parse_port("8080"));
12    // Ok(8080)
13    println!("{:?}", parse_port("abc"));
14    // Err("포트 오류: ...")
15    println!("{:?}", parse_port("0"));
16    // Err("포트는 0이 될 수 없음")
17}
📌 ? 의 동작 원리
expr?는 match expr { Ok(v) => v, Err(e) => return Err(e.into()) }와 동일합니다. .into()가 자동 호출되므로 From 트레이트가 구현되어 있으면 에러 타입이 자동 변환됩니다.
⚠ 반환 타입이 Result여야 한다
?는 함수 반환 타입이 Result(또는 Option)일 때만 사용 가능합니다. main()도 fn main() -> Result<(), Box<dyn std::error::Error>>로 선언하면 ?를 사용할 수 있습니다.
§ 10

map · and_then — 함수형 체이닝

Result도 Iterator처럼 map과 and_then을 지원합니다. 파싱 결과를 연속으로 변환할 때 중첩 match 없이 처리할 수 있습니다.

Rust
 1fn main() {
 2    // map — Ok 값을 변환
 3    let doubled = "21".parse::<i32>()
 4        .map(|n| n * 2);
 5    println!("{:?}", doubled); // Ok(42)
 6
 7    // and_then — Ok → Result 체인
 8    let result = "42".parse::<i32>()
 9        .and_then(|n| {
10            "ok".parse::<i32>()
11                .map(|_| n)
12                .or(Ok(n))
13        });
14    println!("{:?}", result); // Ok(42)
15
16    // 체이닝
17    let v = " 21 ".trim()
18        .parse::<i32>()
19        .map(|n| n * 2)
20        .unwrap_or(0);
21    println!("{}", v); // 42
22}
📌 map vs and_then
map(|v| v*2)은 Ok 값을 변환합니다 — 결과도 Result. and_then(|v| ...)은 Ok 값을 받아 새 Result를 반환합니다. 변환 자체가 실패할 수 있을 때 and_then을 씁니다.
✓ 파이프라인 패턴
input.trim().parse::<i32>().map(|n| n*2).unwrap_or(0)처럼 연결할 수 있습니다. 중간에 Err가 나면 이후 map은 건너뛰고 Err가 전파됩니다.
§ 11

ok() · ok_or() — Option과 Result 전환

Result와 Option은 서로 변환할 수 있습니다. API가 Option을 반환하는데 파싱 결과가 필요할 때, 또는 반대로 변환이 필요할 때 사용합니다.

Rust
 1fn main() {
 2    // Result → Option
 3    let opt: Option<i32> =
 4        "42".parse().ok();
 5    println!("{:?}", opt); // Some(42)
 6
 7    let none: Option<i32> =
 8        "abc".parse().ok();
 9    println!("{:?}", none); // None
10
11    // Option → Result
12    let s: Option<&str> = Some("42");
13    let r: Result<i32, &str> =
14        s.ok_or("값 없음")
15         .and_then(|v| v.parse().map_err(|_| "파싱 실패"));
16    println!("{:?}", r); // Ok(42)
17}
📌 ok()와 ok_or()
result.ok(): Ok(v) → Some(v), Err(_) → None — 에러 정보 버림.
option.ok_or(e): Some(v) → Ok(v), None → Err(e) — 에러 정보 추가.
⚠ ok()는 에러 정보를 버린다
.ok()를 사용하면 Err의 에러 정보가 사라집니다. 에러 원인을 로그로 남기거나 사용자에게 보여줘야 한다면 ok() 대신 map_err으로 변환하거나 match를 사용하세요.
§ 12

여러 필드 파싱 — struct 복원

실무에서는 CSV, 설정 파일, 네트워크 패킷 등 여러 필드를 파싱해 구조체를 만드는 경우가 많습니다. 어느 필드에서 실패했는지 정확히 알려주는 에러 처리가 중요합니다.

Rust
 1fn parse_csv_row(line: &str)
 2    -> Result<(String, i32), String>
 3{
 4    let parts: Vec<&str> =
 5        line.splitn(2, ',').collect();
 6    if parts.len() != 2 {
 7        return Err(format!("필드 부족: {}", line));
 8    }
 9    let name = parts[0].trim().to_string();
10    let score = parts[1].trim()
11        .parse::<i32>()
12        .map_err(|e| format!("점수 오류: {}", e))?;
13    Ok((name, score))
14}
15fn main() {
16    let rows = ["Alice,95", "Bob,abc", "Carol,88"];
17    for row in &rows {
18        match parse_csv_row(row) {
19            Ok((n, s)) => println!("{}: {}", n, s),
20            Err(e)     => println!("오류 — {}", e),
21        }
22    }
23}
📌 map_err로 에러에 컨텍스트 추가
.map_err(|e| format!("점수 파싱 오류: {}", e))?처럼 에러 메시지에 필드 이름을 포함시키면 디버깅이 훨씬 쉬워집니다. 에러가 어느 필드에서 발생했는지 즉시 알 수 있습니다.
✓ splitn으로 안전하게 분리
split(",") 대신 splitn(2, ",")을 사용하면 값 안에 쉼표가 있어도 최대 2개로만 분리됩니다. 필드 수 검증과 함께 사용하면 잘못된 입력을 명확하게 거부할 수 있습니다.
§ 13

커스텀 타입 파싱 — impl FromStr

자신만의 타입에 parse()를 붙이려면 FromStr 트레이트를 구현합니다. 구현하면 parse::<MyType>()를 바로 사용할 수 있습니다.

Rust
 1use std::str::FromStr;
 2use std::num::ParseIntError;
 3
 4#[derive(Debug)]
 5struct Point { x: i32, y: i32 }
 6
 7impl FromStr for Point {
 8    type Err = String;
 9    fn from_str(s: &str)
10        -> Result<Self, Self::Err>
11    {
12        let parts: Vec<&str> =
13            s.split(',').collect();
14        if parts.len() != 2 {
15            return Err("x,y 형식 필요".into());
16        }
17        let x = parts[0].trim()
18            .parse::<i32>()
19            .map_err(|e: ParseIntError|
20                e.to_string())?;
21        let y = parts[1].trim()
22            .parse::<i32>()
23            .map_err(|e: ParseIntError|
24                e.to_string())?;
25        Ok(Point { x, y })
26    }
27}
28fn main() {
29    let p = "3,4".parse::<Point>().unwrap();
30    println!("x={}, y={}", p.x, p.y);
31    // x=3, y=4
32    let dist = ((p.x*p.x + p.y*p.y) as f64)
33        .sqrt();
34    println!("원점 거리: {:.2}", dist); // 5.00
35    println!("{:?}", "a,b".parse::<Point>());
36    // Err("invalid digit ...")
37}
📌 FromStr 구현 구조
type Err 연관 타입으로 에러 타입을 선언하고, from_str(&str) -> Result<Self, Self::Err>를 구현합니다. 내부에서 ?를 자유롭게 사용할 수 있어 복잡한 파싱 로직도 깔끔하게 작성됩니다.
✓ 활용 범위
FromStr를 구현하면 .parse()뿐 아니라 str::parse(), serde의 문자열 역직렬화, clap의 커맨드라인 인자 파싱 등에도 자동으로 사용됩니다.
§ 14

에러 타입 변환 — map_err · From

여러 곳에서 파싱할 때 에러 타입이 제각각이면 ?를 쓸 수 없습니다. map_err로 일일이 변환하거나, From 트레이트를 구현해 자동 변환되게 할 수 있습니다.

Rust
 1use std::num::ParseIntError;
 2
 3#[derive(Debug)]
 4enum AppError {
 5    ParseFailed(ParseIntError),
 6    TooLarge(i32),
 7}
 8impl From<ParseIntError> for AppError {
 9    fn from(e: ParseIntError) -> Self {
10        AppError::ParseFailed(e)
11    }
12}
13fn parse_small(s: &str)
14    -> Result<i32, AppError>
15{
16    let n: i32 = s.parse()?;
17    if n > 100 {
18        return Err(AppError::TooLarge(n));
19    }
20    Ok(n)
21}
22fn main() {
23    let results = ["42", "abc", "999"];
24    for s in &results {
25        match parse_small(s) {
26            Ok(n) =>
27                println!("{} -> Ok({})", s, n),
28            Err(AppError::ParseFailed(e)) =>
29                println!("파싱 실패: {}", e),
30            Err(AppError::TooLarge(n)) =>
31                println!("너무 큰 값: {}", n),
32        }
33    }
34}
📌 From 트레이트와 ? 연산자
From<ParseIntError> for AppError를 구현하면 ?가 자동으로 .into()를 호출해 타입을 변환합니다. 모든 파싱 에러를 하나의 커스텀 에러 타입으로 통합할 수 있습니다.
✓ Box 활용
Box<dyn std::error::Error>를 반환 타입으로 쓰면 어떤 에러 타입이든 ?로 처리할 수 있습니다. 빠른 프로토타이핑이나 main() 함수에서 자주 사용됩니다.
