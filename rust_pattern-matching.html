Rust 패턴 매칭 완전 가이드

The Rust Reference · The Rust Programming Language Book 공식 문서 기반 — 할루시네이션 없이 검증된 내용만 수록. Rust 2021 Edition 기준.

1. 패턴 매칭 전체 구조

Rust의 패턴 매칭은 단순 분기문이 아닙니다. 구조 해체(destructuring), 조건 가드(guard), 변수 바인딩, 소유권 제어까지 통합한 언어의 핵심 기능입니다.

2. 패턴이 사용되는 위치 6곳

공식 레퍼런스 The Rust Reference — Patterns에 따르면 패턴은 다음 6곳에서 사용됩니다.

표현식match 표현식가장 강력한 형태. 컴파일러가 exhaustiveness 강제. 모든 패턴 종류 사용 가능.
표현식if let / if let-elserefutable 패턴 단일 분기. else 선택적. else if let 체이닝 가능.
루프while let패턴이 일치하는 동안 반복. 스택 소비·이터레이터 수동 소비에 자주 사용.
구문let 구문let x = 5도 패턴 매칭. irrefutable만 허용. 튜플/구조체 해체 가능.
루프for 루프for (i, v) in iter — 루프 변수 자체가 패턴. irrefutable만 허용.
선언함수/클로저 인자fn f((x,y): (i32,i32)) 형태. irrefutable만 허용. 클로저도 동일.

3. Refutability — 두 가지 패턴 분류

Irrefutable pattern — 어떤 값이 주어지더라도 반드시 일치하는 패턴.
예: let x = 5, let (a, b) = pair, fn f(x: i32)
let, for, 함수 인자는 irrefutable만 허용합니다.
Refutable pattern — 특정 값에서 일치하지 않을 수 있는 패턴.
예: if let Some(x) = val, if let Ok(n) = result
if let과 while let은 refutable 패턴을 기대합니다.

왜 let에서 refutable이 안 되는가?

let Some(x) = option_val;을 쓰면 컴파일러가 오류를 냅니다. option_val이 None일 때 실행할 코드가 없기 때문입니다. let은 항상 성공해야 하므로 irrefutable 패턴만 허용합니다. 반대로 if let x = val (irrefutable을 if let에 사용)은 컴파일러 경고를 냅니다 — x는 항상 일치하므로 if let이 의미 없기 때문입니다.

4. match는 표현식이다

C·Java의 switch와 달리, Rust의 match는 표현식(expression)입니다. 값을 반환하며 변수에 직접 대입할 수 있습니다.

규칙 1 — 모든 arm의 반환 타입이 동일해야 합니다.
규칙 2 — arm이 블록이면 블록의 마지막 표현식이 반환값입니다 (세미콜론 없이).
규칙 3 — arm에서 return, panic!(), break, continue는 never 타입 !을 가지며 어떤 타입과도 호환됩니다.

예: let grade = match score { 90..=100 => "A", _ => "F" }; — match가 &str을 반환합니다. 일부 arm에서 return을 써도 타입이 맞는 이유가 never 타입 덕분입니다.

5. 패턴 종류 완전 목록

5-1. 리터럴 패턴

정확한 값과 일치합니다. 정수, char, bool, 문자열 리터럴이 가능합니다. 부동소수점 리터럴 패턴은 공식적으로 허용되지만 정밀도 문제로 컴파일러 경고가 발생합니다.

5-2. 범위 패턴

a..=b inclusive range만 패턴에서 허용됩니다. exclusive range (a..b)는 패턴에서 사용 불가입니다 (Rust Reference 기준). 정수형과 char만 지원합니다. i32::MIN..=-1처럼 상수도 끝값으로 쓸 수 있습니다.

5-3. 구조체 해체

  • shorthand — Point { x, y }: 필드명 = 변수명
  • rename — Point { x: a, y: b }: 필드 x → 변수 a
  • .. 무시 — Point { x, .. }: 나머지 필드 전체 무시

5-4. 슬라이스 패턴

  • [a, b, c] — 정확히 3개
  • [first, ..] — 첫 원소만 바인딩
  • [.., last] — 마지막 원소만 바인딩
  • [first, rest @ .., last] — 앞·끝 바인딩 + 중간을 슬라이스로 캡처 (Rust 1.42+)

5-5. @ 바인딩

n @ 1..=10처럼 패턴과 일치하면서 동시에 값을 바인딩합니다. Rust 1.56+에서 구조체 필드 내부 중첩 @도 가능합니다.

6. 패턴과 소유권

패턴 매칭은 소유권 시스템과 긴밀하게 연동됩니다. 이 부분을 정확히 이해하지 못하면 컴파일 오류가 자주 발생합니다.

Rust 2021 match ergonomics — match &val { x => }처럼 참조를 match 대상으로 넘기면 arm의 바인딩이 자동으로 참조가 됩니다. 현대 Rust에서 권장하는 방식입니다. 명시적 ref는 이전 스타일이지만 여전히 유효합니다.

7. 매치 가드 심화

매치 가드(match guard)는 패턴 if 조건 형태로, 패턴이 일치한 후 추가 조건을 검사합니다.

가드는 exhaustiveness 검사에 포함되지 않습니다. 컴파일러는 가드의 조건식을 분석하지 않습니다. 가드가 달린 arm만 있고 _ =>가 없으면 컴파일 오류입니다.

OR 패턴과 가드의 우선순위

A | B if cond에서 가드는 OR 패턴 전체에 적용됩니다. 즉 (A | B) if cond입니다. A | (B if cond)가 아닙니다. 이 차이를 반드시 기억하세요.

가드에서 바인딩 변수 사용

가드 조건에서 arm의 바인딩 변수를 참조할 수 있습니다. 가드가 false이면 바인딩은 이루어지지 않고 다음 arm으로 넘어갑니다. 소유권 이동도 발생하지 않습니다.

8. OR 패턴 규칙

OR 패턴(|)은 Rust 1.53+에서 패턴이 허용되는 모든 위치로 확장되었습니다.

바인딩 일관성 규칙 — OR 패턴의 모든 분기에서 바인딩 변수의 이름과 타입이 완전히 일치해야 합니다.
가능: Ok(x) | Err(x) (이름 같고 x 타입도 일치)
불가: Ok(x) | Err(y) (이름 다름), Ok(x) | Err(_) (한쪽에만 바인딩)

9. let-else 구문 (Rust 1.65+)

let-else는 Rust 1.65에서 안정화된 구문입니다. refutable 패턴이 불일치할 때 조기 탈출(early exit)을 평탄하게 표현합니다.

문법: let PATTERN = EXPR else { DIVERGING_CODE };
else 블록은 반드시 diverge해야 합니다 — return, break, continue, panic!(), 무한 loop 중 하나. 일반 표현식은 컴파일 오류입니다.
바인딩 변수는 let-else 이후 스코프 전체에서 유효합니다 (if let과의 핵심 차이).

10. 중첩 패턴

Rust 패턴은 이론적으로 무한히 중첩 가능합니다. 구조체 안의 열거형 안의 튜플까지 한 번에 해체할 수 있습니다.

중첩 예시:
Shape::Circle { center: Point { x: x @ 0..=100, y }, radius: r @ 1.0..=50.0 }
Circle variant + center.x가 0~100 범위 + radius가 1.0~50.0 인 경우만 일치, x·y·r을 바인딩.

중첩이 3단계 이상이면 가독성이 급격히 떨어집니다. 중간 변수로 분리하는 것을 권장합니다.
